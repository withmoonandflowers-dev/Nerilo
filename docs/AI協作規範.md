# AI 協作規範（Nerilo 專案）

本文件供 **人類與 AI（如 Cursor）協作** 時共用，目標是產出**穩定且符合資安**的產品。

---

## 1. 程式變更後必做

- 執行 **`npm run type-check`**，不得有型別錯誤。
- 執行 **`npm run lint`**，不得新增 lint 錯誤。
- 執行 **`npm run test:run`**，單元測試須全部通過。
- 若變更影響到房間、聊天、等待頁、Firestore：請跑相關 **E2E** 或手動情境驗證。

未通過上述門檻前，**不要合併或上板**。

---

## 2. 資安紅線（絕不允許）

- **禁止**將真實的 API Key、Secret、Token、密碼寫入程式碼或 commit。
- **禁止**將 `.env`、`.env.local`、`.env.production` 加入版控。
- **禁止**在未經人工審核下，變更 Firestore 規則、Auth 權限、環境變數使用方式。
- 正式環境應以**環境變數**提供 Firebase 等設定；程式內 fallback 僅限開發/展示，且需在文件中註明。

---

## 3. 需人工審核的變更

以下類型由 **人** 做最後確認後再合併/上板：

- Firestore 規則（`firestore.rules`）的修改。
- 身分驗證、權限、角色相關邏輯。
- 環境變數的新增或使用方式（含 build/deploy 注入）。
- 依賴套件的新增或升級（需 `npm audit` 與回歸測試）。
- Hosting 安全頭（如 CSP）的調整。

AI 可建議修改內容，但**不得自行合併或部署**上述變更。

---

## 4. 問題修正流程（與 AI 協作時）

1. **重現**：提供清楚步驟或自動化腳本。
2. **分級**：P0（擋上板）/ P1（建議修完再上）/ P2（排期）。
3. **修復**：AI 可產出 patch；人類負責執行 type-check、lint、單元測試與必要 E2E。
4. **合併與上板**：PR 通過 CI 後合併；上板依 [上板與部署手冊](上板與部署手冊.md)。
