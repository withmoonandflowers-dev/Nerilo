# 商業化改造路線圖（架構師建議）

> 推導依據：[GOAL-ANALYSIS.md](GOAL-ANALYSIS.md)。決策記錄：[adr/](adr/)。
> 原則：每個里程碑結束時系統都處於可對外的狀態，不存在「拆到一半」的窗口。

> 2026-07-03 修訂：ADR-0009 定調 Nerilo 為資料傳遞架構（內部平台優先），
> M4 由「看市場再決定」改為明確的平台抽取里程碑。M1 至 M3 不變，
> 因為參考應用（聊天）是平台的第一個客戶與收費驗證場。

## 核心判斷

1. **最大資產**：P2P 訊息不經伺服器的成本結構。競品每則訊息都有伺服器成本，
   Nerilo 的邊際成本趨近於零。這是定價與毛利的護城河，所有改造不得破壞此性質。
2. **最大負債**：宣稱與現實的落差。E2EE 招牌未接電（ADR-0004），
   成本防線只在 client 端（ADR-0005）。前者是信任風險，後者是生存風險。
3. **最大浪費**：72% 的核心程式碼休眠（ADR-0007），維護精力被攤薄。

## 里程碑

### M1：真實性（估 2 至 3 週）→ 2026-07-03 幾近完成

出口條件：對外的每一句技術宣稱都與程式行為一致。

- [完成] ADR-0004：E2EE 接入星型路徑，fallback 改密文，mesh 房間誠實標示
  加密等級（commit 609b34e，@stable E2E 10/10）
- [完成] 重跑安全稽核 critical 項（2026-07-03 逐項核實）：hardcoded 憑證、
  安全 headers、session timeout、logger 均已修；仍開放項為 SDP 簽章
  （移至 M4，與身分統一同批）與信令頻率限制（本來就屬 ADR-0005/M2）；
  匿名自動登入為刻意設計（rules 已擋匿名建房）
- [完成] UX P0 四項核實：Toast、RWD（480/768 斷點）、ConnectionBanner
  （含自動重連倒數）、ErrorBoundary + Skeleton 均已在 3 至 6 月間修復
- [待使用者] 真機兩台煙霧測試（含行動網路，驗證 TURN）——M1 唯一剩餘項

### M2：成本安全（估 2 至 3 週，M1 後）

出口條件：任何單一帳號能造成的最大月成本有數學上限。

- ADR-0006：Blaze 升級與預算斷路器同日上線，部署三個 cleanup 函式
- ADR-0005：rules 總量限制先行，Functions 配額層跟進
- 成本儀表板：Firestore 寫入量、TURN 用量、fallback 觸發率三個指標
  進入每週檢視（fallback 觸發率同時是 P2P 品質指標）
- ADR-0007：休眠模組凍結（第 2 類）與移除（第 3 類）

### M3：收費能力（估 3 至 4 週，M2 後）

出口條件：一個陌生人可以自助付費並獲得 Pro 權益。

- ADR-0008：Stripe（或 Paddle）接入，custom claims 授權，rules 讀 claim 執行配額
- 免費層限額上線，paywall 埋點進 featureLog funnel
- 最小濫用防治：房間內封鎖、檢舉信箱、suspended 帳號機制（與 ADR-0005 共用）

### M4：平台抽取（估 6 至 8 週，M3 後；ADR-0009 已定調此為目的地）

出口條件：一個非聊天的新系統能只依賴傳輸層完成安全、正確、穩定的資料傳遞。

- 傳輸契約文件化：傳遞語義（至少一次 + 端側去重）、加密保證、身分模型
- 正確性接線（GOAL-ANALYSIS GC）：ChatService 發送端附 HLC deps
  讓 CausalOrderingBuffer 端到端生效、啟用 StoreAndForward 離線傳遞、
  MessageAssembler 去重上移至傳輸層、P2P 與 fallback 切換守恆協議
- 身分統一：星型路徑（Firebase uid）與 mesh 路徑（ECDSA identity）
  收斂為單一平台身分層
- 傳輸層自動重連策略（M1 只修了 UI 層的恢復體驗）
- 打包：core 抽為獨立 npm 套件，聊天 app 改為第一個消費者，
  附整合文件與一個非聊天範例（例如協作白板或感測資料流）
- mesh 品質深化（3 人以上場景的 soak test 與混沌測試）
- 契約設計必須內建 ADR-0010 的三個分級維度（頻寬、安全等級、送達模式），
  為 M5 異質傳輸鋪路，避免破壞式變更
- 點數帳本基礎設施（ADR-0011）：伺服器端結算 function、三方簽章收據驗證；
  點數可見不可消費（累積期，收集詐欺樣態）

### M5：異質傳輸（時程另估，M4 後；原則見 ADR-0010）

出口條件：一台接了 Meshtastic 節點的 Chromium 裝置，能與寬頻節點
在同一資料流中交換訊息，安全等級標示誠實。

- 第一批：Meshtastic ISM 頻段經 Web Bluetooth/Serial 接入
  （二進位緊湊編碼、分片、去重上移在 M4 已備）
- 第二批：ham licensed mode（sign-only）、訊息型衛星（計量計費通道）
- IP 衛星（Starlink 類）無需工作，現況即支援
- 點數經濟閉環（ADR-0011）：開放點數折抵訂閱費，通道分級費率上線；
  閘道節點（衛星、LoRa 橋接）的誘因在此驗證

## 依賴關係

```
M1(真實性) ──> M2(成本安全) ──> M3(收費) ──> M4(平台抽取)
```

ADR-0009 已於 2026-07-03 拍板（資料傳遞架構、內部平台優先），
不再是待決依賴。

M1 與 M2 順序不可對調：先讓產品說真話，再開始邀請流量；
配額上線前不做任何形式的推廣（流量等於帳單風險）。

## 明確不做的事

- 不自建 signaling 或 TURN 基礎設施（維持 ADR-0002，Metered.ca 或動態憑證即可）
- 不在 M4 前碰 blockchain 激勵（LocalCreditProvider 維持休眠介面）
- 不追求 20 人以上房間（super-node 拓撲維持凍結，等真實需求）
- 不做原生 app（瀏覽器即產品是 P2P 方案的天然邊界，PWA 是上限）
