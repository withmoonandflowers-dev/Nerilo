# Goal 分析：Nerilo 作為資料傳遞架構（整合版）

> 2026-07-03 v2。取代同日第一版（第一版以「聊天產品商業化」為 G0，
> 產品負責人隨後定調：Nerilo 的目標是成為未來系統的資料傳遞架構，
> 聊天 app 是第一個參考應用。本版以此重建目標樹並整合先前差距分析。）
> 現況事實以 codebase 掃描為準，引用附檔案位置。
> 相關文件：[adr/](adr/)、[COMMERCIALIZATION-ROADMAP.md](COMMERCIALIZATION-ROADMAP.md)

## 0. 定位聲明

Nerilo 不是一個聊天 app，是一層**可重用的 P2P 資料傳遞架構**：
未來系統（不限聊天）透過它傳遞資料，伺服器只負責信令與例外中繼。
聊天 app 的角色是參考應用、試煉場與第一個付費驗證場。
此定位已解決 ADR-0009 的懸問（詳見該文件）。

四大支柱：**安全（GS）、正確（GC）、穩定（GT）、商業誘因（GB）**。
前三者是平台對使用它的系統做出的契約；第四者是平台存在的理由。

## 1. 目標樹

```
G0 Nerilo 成為未來系統可信賴的資料傳遞架構
├── GS 安全：資料在傳遞過程中不可被讀取、竄改或冒名
│   ├── GS1 E2EE 是平台預設，不是應用選配
│   ├── GS2 信令完整性：SDP offer/answer 需簽章防 MITM
│   ├── GS3 身分原語：ECDSA mesh identity 升格為平台級身分層
│   └── GS4 降級不降密：任何 fallback 路徑維持同等加密
├── GC 正確：送出的資料以明確宣告的語義到達
│   ├── GC1 傳遞語義契約：至少一次送達 + 端側去重，白紙黑字
│   ├── GC2 因果順序端到端生效（不是只有接收端準備好）
│   ├── GC3 離線傳遞：接收方不在線時資料不消失
│   └── GC4 傳輸切換守恆：P2P 與 fallback 切換瞬間不掉資料
├── GT 穩定：連得上、斷了會恢復、行為可觀測
│   ├── GT1 P2P 成功率成為一級量測指標與 SLO
│   ├── GT2 重連與降級行為明確且自動
│   ├── GT3 mesh（3 人以上）品質達到與星型同級的信心
│   ├── GT4 平台指標可觀測（延遲、跳數、去重率、fallback 率）
│   ├── GT5 效能有量測基線：延遲與吞吐先量再說，SLO 建立在數據上
│   └── GT6 壅塞緩解：負載可動態分散到多個中繼節點（多路徑）
├── GB 商業誘因：架構本身產生並保護商業價值
│   ├── GB1 成本結構護城河：P2P 傳遞零邊際成本，不可破壞
│   ├── GB2 內部紅利：每個接入的自有系統省下 per-message 費用
│   ├── GB3 外部化路徑：SDK 產品化（按 MAU 計費）
│   ├── GB4 濫用不侵蝕毛利：伺服器端配額使成本有上限
│   └── GB5 中繼經濟：貢獻傳輸資源者獲得對價（過路點數）
│       ├── GB5.1 誘因對齊：中繼的真實成本（電力、頻寬、空中時間、
│       │        衛星費用）由受益者買單，稀缺資源獲得溢價
│       ├── GB5.2 帳本不可偽造：點數真相在伺服器端結算，
│       │        收據由收發雙方簽章
│       ├── GB5.3 兌現路徑：點數折抵訂閱費，不做現金提領
│       ├── GB5.4 反詐欺：sybil、自我中繼、量測灌水有防線
│       └── GB5.5 受益者均攤：中繼成本由資料流的受益方分攤，
│                貢獻與消費在同一本帳上互抵（負擔共享、付費也共享）
└── GP 平台化前提（橫切支撐）
    ├── GP1 傳輸層與應用層解耦（ports 模式已有雛形，需完成切割）
    ├── GP2 參考應用誠實可用（UX P0、宣稱與行為一致）
    └── GP3 維護面積與定位一致（凍結與定位無關的模組）
```

### GS 安全

## 3. GX：異質傳輸擴展（衛星與無線電 mesh）

> 2026-07-03 增補。產品負責人提問：具備相應硬體的設備，能否把
> Satellite Communication 與 Meshtastic / 業餘無線電 mesh 接上這個架構。
> 結論：架構上可整合，但不是「多接一種線路」，而是把平台契約
> 從「單一頻寬等級」升級為「分級傳輸」。原則決策見 [adr/0010](adr/0010-heterogeneous-transport.md)。

### 目標樹增補

```
GX 異質傳輸：資料傳遞契約在寬頻與窄頻、即時與延遲網路上皆成立
├── GX1 傳輸適配層：任何傳輸以 adapter 形式接入，核心不感知底層媒介
├── GX2 窄頻語義：緊湊編碼、分片、空中時間預算，取代「4KB 固定封包」假設
├── GX3 法規相容：每傳輸通道攜帶安全等級標籤（業餘頻段禁加密是法規事實）
├── GX4 DTN 語義：延遲容忍傳遞是一級公民，不是 fallback 的例外情況
└── GX5 離網商業誘因：災防、偏遠作業、緊急通訊市場
```

### 候選傳輸可行性對照

| 傳輸 | 媒介特性 | 接入可行性 | 主要工程落差 |
|---|---|---|---|
| Starlink 等 IP 衛星 | 就是網際網路 | **今天就能用**，零改動（WebRTC 照走，只是 NAT 更依賴 TURN） | 無 |
| Meshtastic（LoRa ISM 頻段） | 每包約 237 bytes、有效速率數百 bps 等級、自帶洪泛 mesh（hop limit）、AES-256 PSK 合法 | **可行**。瀏覽器經 Web Bluetooth 或 Web Serial 連 Meshtastic 節點（Chromium 限定），Nerilo 騎在 Meshtastic 自己的 mesh 之上作為一條窄頻通道，不取代其路由 | JSON envelope 換緊湊二進位編碼 + 分片重組；停用 cover traffic 與 256B padding（空中時間吃不起）；歐規 868MHz 有 1% duty cycle 限制 |
| 業餘無線電（ham 頻段，含 Meshtastic licensed mode） | 頻寬同上或更低；**法規禁止以加密遮蔽內容**（台灣 NCC 與美國 FCC Part 97 同旨），需播報呼號 | **技術可行，安全契約必須降級**：僅簽章不加密（完整性與來源可驗，機密性歸零），UI 必須醒目標示 | 與 GS1「E2EE 預設」直接衝突，解法是 GX3 的每通道安全標籤，不是全域開關 |
| 訊息型衛星（Iridium SBD、Swarm） | 每則 270 至 340 bytes、按則計費、延遲分鐘級 | **可行但貴**：定位為「最後手段」通道，DTN 語義下的最高成本層 | 按則成本與 GB1 零邊際成本紅線衝突，必須計量計費轉嫁（GX5 定價） |

### 與既有支柱的衝突與化解

| 衝突 | 內容 | 化解 |
|---|---|---|
| GX3 對 GS1 | ham 頻段禁加密，E2EE 無法是「所有傳輸的預設」 | 安全契約從布林改為分級：E2EE、僅簽章、明文。通道攜帶標籤，應用層可宣告「本資料流不得走低於 E2EE 的通道」。預設仍是 E2EE，降級必須顯式 |
| GX2 對既有 relay 設計 | Sphinx 4KB 固定封包、cover traffic、padding 全部以寬頻為前提 | 這些屬「寬頻通道專屬」特性，窄頻通道明確不提供匿名性保證（誠實標示，呼應 GP2） |
| GX5 對 GB1 | 衛星按則計費打破零邊際成本 | 紅線改述為「IP 通道零邊際成本不可破壞」；計費通道的成本必須可歸因到使用者並轉嫁 |
| GX4 對 GT1 | P2P 成功率指標假設「連上或 fallback」二分 | 指標泛化為每通道等級的送達率與延遲分布 |

### 對既有資產的再評價

DTN 場景讓休眠模組的價值再次上修：StoreAndForward（GC3）、
CausalOrderingBuffer 與 HLC（無需時鐘同步，正是斷續網路要的）、
MessageAssembler 去重（洪泛 mesh 必備）都是異質傳輸的直接載體。
第一版分析要丟、第二版列為戰略資產、本版確認它們是 GX 的核心依賴。
反之 relay/ 的匿名層（Sphinx、cover traffic）確認為寬頻專屬。

### 效能現況與壅塞緩解（GT5、GT6，2026-07-03 增補）

程式碼層面的硬事實（皆有檔案佐證）：

| 路徑 | 機制 | 理論特性 |
|---|---|---|
| 星型（2 人） | WebRTC DataChannel 直連，64KB backpressure 佇列（P2PChannelBus.ts:23） | 延遲即網路 RTT（區域內數十 ms）；聊天負載遠低於瓶頸；大負載會先撞 JSON+Base64 編碼與主執行緒，不是撞頻寬 |
| mesh 3 至 5 人 | full mesh，fanout=n-1、ttl=1（AdaptiveTopologyManager.ts:61） | 單跳，延遲與星型同級 |
| mesh 6 至 20 人 | partial mesh，fanout=3、ttl=3 至 4 | 延遲 = 跳數 × RTT；洪泛放大由去重抑制 |
| 20 人以上 | fanout=4、ttl=5 加 super-node 選舉 | 已有程式碼，未接線（ADR-0007 凍結類） |
| Firestore fallback | 每則一次寫入 + listener | 延遲數百 ms；上限是費用不是技術 |

**尚無 production 量測基線**：MetricsCollector 具備 msgs/sec、延遲直方圖、
跳數分布能力但 opt-in，正式環境沒開。GT5 的第一步是把基線量出來，
不是先優化（M2 儀表板工作的一部分）。

**多節點動態分流（GT6）已經寫好了，在休眠**：MultiPathSelector 支援
2 至 4 條不相交路徑（≥3 個候選節點時啟用，MultiPathSelector.ts:18-22）、
PathQualityTracker 以 50 樣本滑動視窗回饋路徑品質、RelayScorer 按
延遲/可靠度/頻寬/在線率評分選節點、MessageAssembler 首達去重。
這正是「多節點同時支援緩解塞車」的完整實作，屬 ADR-0007 第 1 類
待啟用資產，接線排 M4（廣頻）/M5（異質通道）。

### GX 與 GB5 的互相成全

異質傳輸讓過路點數第一次有了真實的經濟基礎。在純 WebRTC 世界，
中繼的邊際成本近零，點數只是遊戲化；在 GX 世界，閘道節點
（衛星橋接、LoRa 橋接）承擔真金白銀的成本（衛星按則計費、
空中時間預算、行動裝置電力），沒有對價就沒有人願意當橋。
反過來，點數也解決了 GX5 的計費難題：Nerilo 不必對每種通道
做計量帳單，而是讓網路內部以點數清算，平台只在點數與訂閱費的
兌換點收口。詳見 [adr/0011](adr/0011-relay-credit-economy.md)。

## 4. 整合結論

**定位翻轉改變了休眠程式碼的價值判斷。** 第一版分析建議把 transport/、
ordering/ 等模組移出主幹；在平台定位下，它們正是 GC（正確性）支柱的
實作載體，改列為待啟用資產。game/、community/ 依然與定位無關，維持凍結
或移除（ADR-0007 修訂版）。

**阻塞鏈不變，終點改變**：
GP2 誠實（M1）→ GB4 成本安全（M2）→ 收費驗證（M3）→ **GP1+GC 平台抽取（M4）**。
前三個里程碑原樣成立，因為參考應用是平台的第一個客戶與付費實驗場；
M4 從「看市場再說」升格為路線圖的目的地。

**四支柱的張力點**，設計時需明知故選：
- 安全與正確互斥處：E2EE 之後伺服器端無法去重與排序，GC 的所有機制必須在端側完成（這是架構原則，不是實作細節）。
- 穩定與成本互斥處：fallback 越可靠，Firestore 帳單越高；GT2 的降級策略必須以 GB4 配額為邊界。
- 商業誘因與開放互斥處：SDK 外部化（GB3）後，配額與計費是唯一收入閘門，GS/GC 契約則是唯一信任來源，兩者都不能靠「先上再補」。
