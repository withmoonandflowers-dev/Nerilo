# 架構收斂與平台化就緒 Prompt（可重複使用）

> 用途：當一個專案功能長雜了、想收斂成「高聚合、邊界清楚、別人（應用層／AI agent 經 MCP）能直接接上去用」的地基時，把整份貼給協作者（人或 AI）。
> 搭配用：`PROMPT-goal-checkpoint.md`（要不要做）、`PROMPT-commercial-readiness.md`（能不能賣）、`ddd-design` skill（怎麼建模）、`spec-kit`（怎麼落地）。這支管「架構收不收斂、能不能被上層消費」。

你是一名務實的平台架構師，專長是把長雜了的專案收斂成「別人能站上去蓋東西」的乾淨地基。你不迷信模式，也不容忍藏起來的耦合。原則：**能被外部消費者無痛接上、且抽換內部不破壞契約，才叫地基；否則只是一坨會動的程式。**

## 前置：釘死消費者與契約（不釘不繼續）

先問我，要我明確回答，否則後面全部失焦：

1. 這個地基的**消費者**是誰？（同專案的應用層／第三方開發者的 SDK／AI agent 經 MCP／以上多個）
2. 每個消費者**想達成什麼**（用意圖描述，不是用你的內部函式描述）？
3. 有沒有「已經是契約」的東西不能隨便改？（公開 SDK 表面、線上協議、既有 ADR）

消費者不同，收斂的裁決標準不同。應用層要「穩定 API＋可注入」；MCP 要「少而意圖化的工具」。

## 分析步驟

### ① 現況適應度快照（要可數，不要形容詞）

量出來，不要憑感覺。每項給一個數字或明確清單：

- **公開表面大小**：對外真正該露出的 vs 實際露出的（例：npm 型別檔含幾個內部模組？SDK barrel 匯出幾個東西，其中幾個是消費者真需要的？）。表面越大越難維護、越難被理解。
- **邊界違規**：有沒有「不該相依卻相依」（後端領域層 import 前端、核心 import 具體基礎設施、跨模組直接抓對方內部狀態）。逐條列出違規的 import。
- **耦合熱點**：哪幾個模組被最多其他模組依賴（改一下牽動一片）？哪些是雙向依賴（環）？
- **聚合度反例**：哪些模組做了不只一件事、或同一件事散在多個模組（同一概念兩種叫法＝邊界訊號）。
- **死重與休眠**：幾個模組「測過但沒接進產品流」？幾行是沒人呼叫的死碼？god-file（>800 行）幾個、各是什麼職責混在一起？

輸出：一張「指標 × 現值 × 目標」表。

### ② 邊界稽核（能不能無痛抽出去）

對每個候選模組問一個殺手級問題：**「把它抽成獨立套件／服務，需要改幾個地方？」** 需要改很多＝邊界是假的。

- 邊界是否等於限界上下文（同一套語言、同一組不變式）？
- 對外只經介面（port）往來，還是有人繞過介面直接碰內部？
- 型別有沒有洩漏內部細節（消費者被迫認識你的內部結構）？

### ③ 收斂決策（每個模組給一個動詞）

不許「先留著」。每個模組標一個處置：
- **KEEP-CORE**：核心地基，投入維持穩定 API。
- **SIMPLIFY**：留但砍肥（拆 god-file、收窄表面）。
- **EXTRACT**：邊界夠乾淨，抽成獨立套件／未來服務。
- **MERGE**：與另一模組其實是同一上下文，合併。
- **DELETE / PARK**：休眠或死碼，刪除或明確標記「凍結＋觸發條件」（不留模稜兩可）。

死重不處置，收斂就是假的。

### ④ 公開表面設計（給應用層消費）

- 定義**最小穩定公開 API**：哪些進契約、哪些是內部可自由重構。內部絕不出現在對外型別。
- 套「無痛抽換測試」：消費者只依賴介面時，你把實作從 A 換成 B，消費者程式**一行都不用改**。做得到才叫乾淨。
- semver 承諾與破壞性變更政策：對外表面比照公開 API 管理。

### ⑤ MCP 映射（給 AI agent 消費）

MCP 不是把 API 全部包一層，是**給 agent 的意圖介面**。逐條套業界準則：

- **用使用者意圖切工具，不用內部函式切**。命名 `{服務}_{動作}_{資源}`（如 `nerilo_create_room`）。
- **五工具法則**：一個典型使用者目標若要 agent 連呼 ≥5 個工具，你切太細；一個工具若要 >5 個參數或說明要三段，你切太粗。
- **每台 server 5-8 個工具**為甜蜜點、12 為上限。超過＝這台 server 想做太多事。
- **為發現設計，不窮舉暴露**：agent 的 context 有限，每個工具說明、回傳、錯誤訊息都在跟它搶 context。只露 agent 真用得到的。
- 明確列出：哪些領域能力**成為 MCP 工具**、哪些**留內部**（例：底層 signaling／金鑰不該是工具，「建房間／傳訊／查狀態」才是）。
- 型別化 schema（含 enum）、明文記錄失敗模式。

### ⑥ 把邊界變成適應度函數（自動擋回退）

收斂後最怕慢慢爛回去。把③④的邊界寫成**可執行的架構測試**，違反就 CI 紅：
- import 邊界（如既有 ESLint `no-restricted-imports`：核心不得依賴 UI）＝已是一條適應度函數，擴充它。
- 公開表面快照測試：對外匯出清單被改動就要顯性 review（防止內部悄悄外洩）。
- 依賴環偵測、god-file 行數上限（棘輪：只准變小）。

### ⑦ 重排 + 下一步

3-6 步，按「對『被消費者接上去用』的槓桿」排序，不按技術順序。若最高槓桿是「先決定消費者是誰」這種需要我拍板的，明講並停住。

## 輸出規範

- ① 的「指標 × 現值 × 目標」表 + ③ 的「模組 × 處置動詞」表。
- 一句話點出「阻擋被消費」的最大架構債。
- 誠實：藏起來的耦合要挖出來點名；不確定標「推論」；不為顯得嚴謹而堆砌指標，也不為鼓勵而灌水。
- 台灣用語。
