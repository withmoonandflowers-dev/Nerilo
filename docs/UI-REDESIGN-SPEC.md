# UI 重設計規格（Vue 版）— iMessage / Telegram 亮色通訊風

> 2026-07-04。隨 ADR-0017 產出。目的：把「美」定義成可驗收的具體契約，
> 每一頁的設計都必須能回答「這在 iMessage/Telegram 裡長什麼樣」。
> Goal 對應：GP2 參考應用誠實可用——試煉場必須可示人。

## 1. 視覺語言（設計 token 契約)

| Token | 值 | 參照依據 |
|---|---|---|
| 底色 | `#F2F2F7`（iOS systemGroupedBackground） | iMessage 列表底 |
| 卡片/氣泡面 | 純白 `#FFFFFF` | iMessage 對話列 |
| 主色（我方氣泡/CTA） | `#0A84FF` iOS 藍 | iMessage 藍 |
| 輔助面（對方氣泡） | `#E9E9EB` | iMessage 灰氣泡 |
| 文字主/次 | `#000000` / `#8E8E93` | iOS label/secondaryLabel |
| 成功/警示/危險 | `#34C759` / `#FF9F0A` / `#FF3B30` | iOS system colors |
| 圓角 | 氣泡 18px、卡片 16px、按鈕 12px、輸入框 20px（膠囊） | iMessage 幾何 |
| 字體 | system-ui 堆疊（蘋果機自動 SF Pro） | 原生感的最大來源 |
| 陰影 | 極輕：`0 1px 3px rgba(0,0,0,.06)`；靠底色對比分層，不靠重陰影 | iOS 分層邏輯 |
| 動效 | spring 曲線 `cubic-bezier(.34,1.56,.64,1)`，180–280ms；氣泡進場 scale+fade | Telegram 訊息進場 |
| 深色 | 保留一組 dark（`#000` 底、`#1C1C1E` 面），跟隨系統 | iOS 深色模式 |

禁止事項（上一輪「醜」的來源清單）：漸層大色塊當底、工程感藍紫 `#667eea`、
細碎邊框線、密集小字表格式排版、無意義的裝飾動畫。

## 2. 功能 → 畫面映射（goal 導向）

### /login — 進門即懂
- 單卡置中、app 名稱 + 一句話定位（「點對點加密聊天」）。
- Google 登入為主 CTA（膠囊大按鈕），匿名體驗為次要文字鏈。
- Email 表單收進「使用 Email」展開區，不佔首屏。
- Goal：降低第一次使用門檻（GP2）；登入方式的視覺權重 = 轉換率排序。

### /dashboard — 像通訊 app 的「對話列表」，不是管理後台
- 主體是房間列表（每列：房名、成員數徽章、相對時間、chevron），
  點列直接進房——對齊 iMessage 對話列心智模型。
- 建房是右上角「＋」，彈出 sheet 式 modal（房名 + E2EE 開關），
  不再是佔半屏的常駐表單。
- 加入房間（貼碼）收進「＋」sheet 的第二個 tab。
- 空狀態：置中插圖 + 「建立第一個房間」主按鈕。
- Goal：把「管理工具感」轉成「通訊 app 感」——這是審美判定的核心差距。

### /waiting/:roomId — 等待室即邀請頁
- 大 QR code 卡片置中（分享是等待室唯一任務）、
  複製連結膠囊按鈕、Web Share 整合。
- 成員頭像列（圓形、上線即彈入，spring 動效）。
- 房主見主 CTA「開始聊天」；成員見「等待房主開始…」脈動指示。
- Goal：縮短「建房 → 對方入房」的社交摩擦（GT1 的前置漏斗）。

### /chat/:roomId — 主戰場，1:1 對齊 iMessage
- 氣泡：我方右側藍底白字、對方左側灰底黑字，18px 圓角
  （尾端那一角 4px，做出 iMessage 的「尾巴」方向感）。
- 連續同人訊息群組化（間距 2px vs 換人 12px），只在群組尾顯示時間。
- 送達狀態：氣泡下方小字「傳送中 / 已送達 / 失敗（點按重送）」，
  僅顯示在最後一則自己的訊息（iMessage 規則）。
- 輸入列：膠囊輸入框 + 圓形藍色送出鍵（有字才亮），自動增高至 4 行。
- 連線狀態：頂欄副標（「已連線 · P2P 直連」/「重新連線中…」），
  斷線時頂部橫幅 + 重試按鈕；E2EE 徽章 = 標題旁小鎖。
- 新訊息浮動按鈕「↓」帶未讀數。
- Goal：GC/GT 的使用者可見面——語義（送達/失敗/加密）誠實呈現（GP2）。

## 3. 資訊架構不變

路由、Firebase 資料流、P2P 初始化順序與 React 版完全相同——
本次只換皮與框架，不改任何行為語義，E2E 改寫時 flow 步驟可沿用。

## 4. 驗收標準

1. 五分鐘盲測：截圖給不知情者看，第一反應是「通訊軟體」而非「管理系統」。
2. 每頁與本規格逐條對照，偏離需有書面理由。
3. 產品負責人過目四頁截圖後拍板，才進入 E2E 改寫與切換。
