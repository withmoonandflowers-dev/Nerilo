# ADR-0012：社群自營中繼基礎設施（三層備援階梯）

- 狀態：Proposed（原則現在定，第 1 層 M4、第 2 層 M5）
- 日期：2026-07-03

## Context

現行唯一備援是 Firestore relay：每則訊息一次計費寫入，延遲數百 ms，
「上限是費用不是技術」。營運者獨自承擔所有備援成本，
流量成長等於帳單線性成長，與單人營運的資源不相稱。

產品負責人提問：能否不完全依賴營運者，讓社群自己建立備援容量。

架構條件盤點：
- E2EE 已接電（ADR-0004）：中繼節點搬運的是密文，content-blind，
  社群節點**無須被信任**即可承擔中繼。這是可行性的密碼學前提。
- 中繼與多路徑機制已實作在休眠（RelayManager、MultiPathSelector 2 至 4
  不相交路徑、PeerScoring、MessageAssembler 去重）。
- 點數經濟已設計（ADR-0011）：貢獻中繼賺點數、受益者均攤、
  閘道營運者以訂閱全免加額度作為對價。
- 成本結構事實：信令是小額寫入（便宜），中繼流量才是成本大頭。

## Decision

1. **備援從單一 Firestore 改為三層階梯**，依序嘗試：
   - 第 1 層 peer 中繼：線上瀏覽器節點經 WebRTC 相互中繼（M4 接線
     休眠的 relay 層）。解當下線上的壅塞與直連失敗。
   - 第 2 層社群常駐節點：headless daemon（Node runtime），任何人可跑
     （樹莓派、VPS），提供常駐中繼與離線收件匣，依 ADR-0011 賺點數。
     原 adapters/ 不復用（型別殼，ADR-0007 維持移除），M5 以真實 daemon
     需求重寫 runtime 層。
   - 第 3 層 Firestore：最後手段保底。目標狀態下承擔的是尾端流量，
     營運者成本從「隨流量線性」轉為「近乎固定」。
2. **營運者永久保留的兩件事**：信令（房間發現與 SDP 交換）與點數帳本
   結算。兩者都便宜且是信任錨點；中繼與儲存容量則完全開放社群供給。
3. **不信任社群節點，用機制補**：多路徑投遞使單一惡意節點無法阻斷送達；
   PeerScoring 降權壞節點；三方簽章收據（ADR-0011）使記假帳需合謀。
   社群節點看不到明文（E2EE），也不掌握身分真相（帳本在營運者端）。
4. **啟動順序刻意為「激勵先行」**：M4 先讓點數帳本與 peer 中繼上線，
   讓「跑節點有報酬」成為事實之後，M5 才發布 daemon。冷啟動期
   Firestore 維持主力，其成本由 ADR-0005 配額限制上界。
5. **完全去中心化（信令也下放、DHT 發現）不在本 ADR 範圍**：
   那會移除收口點與帳本信任錨，動搖 GB3/GB4 的商業結構，
   若未來要走，必須開新 ADR 並重新評估整個經濟模型。

## Consequences

- 營運者成本模型質變：Firestore 從成本中心降為保底，社群容量隨用戶成長自然擴容（點數驅動），單人營運的可持續性大幅提高。
- 出現新的軟體交付物：headless daemon（M5），含打包、自動更新、節點健康監控，維護面積增加；以「社群節點 = Pro 免費」的對價換取。
- 塞車緩解（GT6）與成本轉移合一：多路徑的每條路徑收據都進均攤，社群節點越多，壅塞與帳單同時下降。
- 離線投遞的隱私語義變化：訊息密文暫存在社群節點而非 Firestore，保密性不變（E2EE），但可用性從 Google SLA 變成社群節點的盡力而為，多副本（存多個節點）是 M5 的必要設計。
- 冷啟動風險：若點數誘因不足以催生節點，第 2 層長不出來，架構退化為現狀（Firestore 主力）。此風險由 M4 累積期的數據（有多少用戶願意開 peer 中繼）提前驗證。
