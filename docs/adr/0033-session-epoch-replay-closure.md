# ADR-0033：session epoch 入簽章，以現行代門檻收斂跨會話重放（R1 閉環）

- 狀態：accepted（2026-07-18）
- 關聯：Spec 009、ADR-0026（R1 短期接受排修 P1）、ADR-0023（P1 複本持久化）、QA-REPORT 已知限制第 1 項

## 決策

1. **per-sender 自宣告會話代**，不做房間層共識代。mesh 無權威節點，房間層代的協調與拜占庭成本不成比例；自宣告代靠簽章不可偽造（採納只發生在驗簽＋身分綁定之後），攻擊者能重放的最高代就是 sender 真實現行代，重放它等於正常 gossip。
2. **配發規則取 max(持久下一值, Date.now())**。這是「清儲存／換裝置 epoch 歸零會把 sender 鎖死」的解：新裝置首代由當下時鐘種出，正常時鐘下必然高於任何舊代（舊代由更早的時鐘種出）。代價：時鐘倒退＋清儲存並存時 sender 暫時被拒收（fail-closed 方向，不破壞不變量），文件揭露。
3. **現行代門檻、無補送豁免通道**（使用者拍板，比 spec 原列選項更嚴）。放棄「跨會話補歷史」產品能力，換取「舊內容重現」與「預佔槽位」同時堵死且協議面最小。同代內 anti-entropy 補送完好：epoch 門檻與 wall-clock 正交，30 分鐘補送回歸鎖不動。
4. **去重鍵與 digest 全面分代**：(senderId, sessionEpoch, seq)；digest 每 sender 只宣告現行代。宣告舊代是徒勞（收端必拒），只會浪費補送頻寬。
5. **strictProtocols 相等語義、無過渡期**：v1/v2 本質不互通（序列化不同、驗簽必敗），任何降級通道都是把防護送給攻擊者剝除；選 fail-visible 提示雙方更新。
6. **legacy 紀錄標 sessionEpoch=0、只供本機、不重簽不補送**：舊簽章不覆蓋 epoch，無法誠實升格；聊天歷史顯示本來就存應用層 chatStorage，複本層 inert 化不影響使用者可見歷史。
7. **信使回填順手收緊**：runCourierBackup 的 ingest 原本直寫持久層不驗簽，是與本 spec 同構的注入面；一併加驗簽＋身分＋epoch 門檻。

## 誠實殘留

- 全新 store 收端在得知某 sender 現行代之前，舊會話重放仍會被當現行接受（有界：見到更高代即拒後續；佔位在任何情況下結構上不可能）。這是自宣告代的固有限制，完全消除需外部權威或成員間代目錄，皆屬未來項。
- 多裝置同身分的 epoch/seq/floor 同步未解（既列獨立優先項）。
