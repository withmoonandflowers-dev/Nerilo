# ADR-G01：以 Nerilo 為個人遊戲的底層資料串流

- 狀態：Accepted（前提假設：小型合作/回合制遊戲）
- 日期：2026-07-04
- 相關：ADR-0015（遊戲為第二應用）、ADR-0018（wire 格式）、ADR-0019（通道分流）、docs/INTEGRATION-SPEC.md

## Context

評估把個人/獨立遊戲的多人資料串流建在 Nerilo（P2P 傳輸底層）上。Nerilo 提供
DataChannel（可靠控制 + 不可靠狀態雙通道）、E2EE、NAT 穿透、二進位 codec、
lockstep SDK（InputBuffer/seed 協商/state-hash/rollback 原語）、房主接替。

## Decision

**採用 Nerilo，但限定在其甜蜜區的遊戲類型。** 適用邊界是本決策的核心——
用錯類型 Nerilo 會變天花板。

### 適用（GO）
- 回合制（棋、卡牌）、合作、朋友間、小團體 2–5 人、同區域、慢~中節奏、休閒。

### 不適用（NO-GO，需商業方案如 Photon/Steam 網路）
- 競技/排名、快動作（格鬥/FPS）、大人數（6+）、跨區吃反應、需公平性保證。

### 決策依據（優劣權衡）

| 優勢 | 說明 |
|---|---|
| 近零伺服器成本 | 玩家資料直傳，不經付費伺服器；人多不會破產 |
| netcode 底層已完成且測過 | NAT/重連/雙通道/二進位/版本協商都在，只寫遊戲邏輯 |
| lockstep 硬功夫已備 | InputBuffer、seed commit-reveal、state-hash、rollback 原語 |
| 高頻狀態流正確形狀 | 不可靠 state 通道 + 二進位，頻寬非瓶頸 |

| 劣勢（決定能不能用） | 對策 |
|---|---|
| **無伺服器權威 → 作弊擋不住** | 限合作/回合制/朋友間；競技不做 |
| 8~15% 配對 NAT 連不上 | TURN（換成自己付頻寬）或社群中繼（未接線）；接受少數連不上 |
| 跨區延遲 200ms+ + lockstep 輸入延遲 | 限慢~中節奏、同區域優先 |
| 人數天花板（mesh O(n²)） | 限 2–5 人 |
| 只有傳輸，無配對/排行/存檔/反作弊 | 自建，或借 Firebase 帳號/房間 |
| **SDK 從未跑過真實遊戲** | 你是第一個真實使用者，會踩整合雷；接受並回報 |

## Consequences

- **好處**：省成本、netcode 免從零、可當 Nerilo 第一個真實 dogfood + 補助展示。
- **風險/成本**：類型選錯即卡死（作弊/延遲/人數）；SDK 首次實戰有未知 bug；
  無社群支援，bug 只有自己扛。
- **不變式**：遊戲層負責遊戲邏輯、配對、UI、（若需要）反作弊；Nerilo 只負責
  「安全把 bytes 送到對端」。權責邊界見 C4 與整合規格。
