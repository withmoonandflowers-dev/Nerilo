# Demo 腳本：三分鐘演示「斷線也送得到」

主軸：盲信使離線送達。輔線：端到端加密（伺服器只見密文）。
環境：本機 dev server（http://localhost:3000）或 production（https://nerilo.web.app）。
建議先在本機彩排一次，正式場合用 production 網址更有說服力。

## 事前準備（demo 前 10 分鐘）

1. 開三個獨立瀏覽器視窗（建議一般視窗 + 兩個無痕，避免登入態互踩）：
   - 視窗 A：寄件者
   - 視窗 B：收件者
   - 視窗 C：信使節點（掛在 dashboard 即可，不用進房）
2. A、B、C 各自登入不同帳號。C 停在 dashboard（信使節點隨 dashboard 常駐）。
3. A 建房、B 加入，先互發一兩句訊息確認連線正常（狀態列顯示已連線與加密指示）。
4. 準備好 Firestore console 分頁（選用，用於展示密文）。

## 演示流程

### 第 0 到 1 分鐘：正常通訊 + 加密證明

- A 與 B 互發訊息，指出：訊息即時到達、已讀回報、加密指示燈。
- 講一句話：「這些訊息走裝置對裝置直連，伺服器只幫忙牽線。」
- （選用）切到 Firestore console，展示備援集合裡只有 ciphertext 與 iv 欄位，
  沒有任何可讀內容。這是「伺服器只見密文」的直接證據。

### 第 1 到 2 分鐘：關鍵一幕，收件者離線

- 把 B 的視窗整個關掉（不是切分頁，是關掉，口頭強調「他現在完全離線」）。
- A 繼續發兩三則訊息。指出 A 這邊照常送出。
- 講解：「訊息現在以密文寄放在 C 這種路過的陌生節點上。C 驗得出這是真的訊息，
  但打不開內容。」（可指 C 的 dashboard，信使節點正在背景運作。）

### 第 2 到 3 分鐘：回線領取 + 互助點數

- 重新開 B 的視窗、登入、進同一個房間。
- B 離線期間的訊息陸續出現，順序正確、不重複。這就是「恰好一次」。
- 指出 C 的點數變化：信使送達後憑雙方共簽的收據賺到點數，「幫別人傳訊息是有回報的，
  這讓網路能自己長大」。
- 收尾一句：「整套能力已封裝成開源 SDK，其他公益專案可以直接嵌進自己的系統。」

## 常見問題備答

- 問「跟 LINE 差在哪」：LINE 斷網就斷，且平台看得到 metadata。Nerilo 在基礎設施受損時
  仍能透過節點互助送達，內容全程加密。
- 問「信使會不會偷看」：技術上不可能，信使拿到的是密文，簽章可驗真偽但無金鑰可解。
- 問「怎麼防止刷點數」：收據需寄件方與信使雙方簽章，單方偽造不成立，另有賺點速率上限。
- 問「安全性誰背書」：附核心不變量稽核報告（docs/audit/），已知風險全數揭露並有處置紀錄。

## 失敗備援

- WebRTC 連不上（場地防火牆）：改用手機熱點；或退而演示本機環境。
- B 回線沒收到：等 anti-entropy 對帳週期（數秒），口頭解釋補送機制正在工作；
  超過 30 秒未到則重整 B 頁面。
- 全部炸掉：播放事先錄好的螢幕錄影（demo 前務必先錄一份成功版本）。
