# Nerilo 工程可信度說明

供補助與競賽評審、以及潛在採用者評估用。這份文件回答一個問題：Nerilo 的技術宣稱，有沒有可查核的證據撐得住。原則和定位文件一致，只陳述守得住的部分，弱點與已知限制照實列出。

## 一句話

Nerilo 是瀏覽器原生、點對點、端到端加密的韌性通訊層。它的技術可信度不建立在口號上，而建立在四件可被外部查核的東西：一份以嚴格框架執行並標明「有條件通過」的核心不變量技術稽核、一份公開的風險登記表、二十八篇架構決策紀錄，以及一千三百多個會在每次改動時執行的自動化測試。原始碼採 Apache-2.0 授權公開，任何人都能自行驗證下述每一項。

## 四條核心承諾，以及各自怎麼被守住

系統把自己的正確性收斂成四條不變量，每一條都有對應的機制與測試，不是文件上的形容詞。

1. 訊息恰好一次送達。做法是先保留序號再送出，房間金鑰加密、簽章覆蓋密文、八卦式廣播，收端驗簽並以寄件者加序號去重，連線不足時每兩秒一次的對帳補送缺漏。這條在連通的節點圖上數學上保證最終一致，並以三人網狀端到端測試連續多批驗證，矩陣每格恰好一次。

2. 端到端機密性。身分用 ECDSA P-256，金鑰交換走 ECDH，內容用 AES-256-GCM 的房間金鑰加密並隨成員變動輪替金鑰世代。伺服器只經手密文，看不到內容。

3. 點數帳本正當性。中繼賺點必須有雙方共簽的收據，收據經雙簽驗證後才入雜湊鏈帳本，防止單方偽造。

4. 身分與授權。節點識別碼由公鑰雜湊而來，寄件者身分與公鑰綁定，資料庫規則以每把金鑰的差異授權寫入，非成員無法冒充。

## 可查核的證據

以下每一項都在公開的原始碼庫裡，可自行核對。

- 核心不變量稽核報告（`docs/audit/`）。以「錯了等於真實損失」的標準檢視四條不變量，最終結論為有條件通過，無 Critical 級發現。報告同時附上機器可讀的風險登記表與依實際程式碼繪製的架構圖。
- 公開的風險登記表（`docs/audit/core-invariants-risks.csv`）。十項風險逐條標明嚴重度、所在模組、影響、對應的不變量與建議處置，其中無 Critical、兩項 High、六項 Medium、兩項 Low。把弱點攤開，是這份文件可信的前提。
- 二十七篇架構決策紀錄（`docs/adr/`）。每一個重要取捨，包含為何選 A 不選 B，都留有紀錄，決策可追溯。
- 自動化測試。單元測試一千三百八十六個、React 與 Vue 兩套端到端測試、以固定的黃金資料回歸的關鍵流程，另有變異測試與屬性測試對部分核心模組加壓。改動運作中的路徑前，流程要求先寫測試釘住現況行為，再改。
- 常設架構文件（`docs/architecture/`）。以 C4 與領域驅動設計的語言描述系統，含七張依實碼繪製的圖。

## 工程紀律

可信度不只看有沒有測試，更看團隊怎麼面對自己的錯。這個專案有幾個可觀察的習慣。

架構決策先寫紀錄再動手，邊界靠工具強制而非靠自律。後端領域層不得相依前端介面層這條規則，由 lint 規則擋住，違反就無法通過檢查。系統把自己的已知邊界寫在程式註解與決策紀錄裡自我揭露，而不是藏起來。測試流程有一條明文鐵律，不准為了讓測試變綠而放寬逾時，逾時是真訊號。這些做法讓外部查核者能相信報告裡寫的與程式碼裡做的一致。

## 誠實揭露的已知限制

以下限制在稽核報告中列明，此處一併說明處置狀態，不迴避。

- 加密降級的明文相容模式（原列 R2）。金鑰交換失敗時系統原本會靜默退回明文。已改為明確可見：明文房需使用者明確確認才送出，預設拒送，介面顯示真實加密狀態。
- 跨會話重放（R1）。持有舊簽章訊息者理論上可在新會話重放。處置方向為把會話世代納入簽章，列為待辦，尚未落地。
- 中繼賺點未在型別層強制收據參照（R5）。正當性目前在上游層把關，型別層的強制列為改善項。
- 中繼資料未加密（R6）。時間戳、成員、序號等中繼資料在伺服器端為明文，理論上可做社交圖分析。已文件化於威脅模型，長期方向為中繼資料最小化。

把這些寫出來不會削弱可信度，反而是可信度的來源。一份只有優點的技術文件，通常代表沒有被認真檢視過。

## 開源與可查核性

原始碼採 Apache-2.0 授權，包含明確的專利授權，對想嵌入的採用者友善。整個庫公開，上述稽核報告、風險登記、決策紀錄、測試全部可讀。任何評審或採用者不需要相信這份文件，可以自己打開來看。

## 目前還缺什麼

技術與稽核這塊已到可對外查核的程度，但信任資產還有一塊沒補齊，就是第一個真實使用者與在地防災場景的落地驗證。這是下一階段最該投入的地方，也是這份文件不會誇稱已完成的部分。
