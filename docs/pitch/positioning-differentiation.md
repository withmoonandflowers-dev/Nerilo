# Nerilo 定位與競品差異化

供提案、對外簡報與內部決策共用。原則：只講守得住的，不迴避對手領先的地方。

## 一句話定位

Nerilo 是一層「可嵌入的韌性通訊技術」，不是又一個聊天 App。別人蓋災防、匿名通報、社區協作的系統時，底層可以直接串接 Nerilo，得到端到端加密、斷網送達、點對點直連的能力，而不需要自己實作密碼學與 P2P。

## 我們不是什麼，是什麼

不是：一個要跟 Signal、WhatsApp、LINE 搶使用者的加密聊天 App。在那個戰場，資源比我們多百倍的對手已經在，正面打會輸。

是：韌性通訊的「基礎能力層」。定位參照通訊界的 Stripe 或 Twilio。使用者面向的產品是別人的，Nerilo 是他們底下那一層。

這個定位是刻意選的。它讓我們避開紅海，落在一個目前沒有人佔的位置：把「斷網也能傳的加密通訊」做成別人能嵌進自己系統的 SDK。

## 競品對照

以下對照聚焦「災害韌性」與「可被第三方嵌入」兩個 Nerilo 真正在意的維度。誠實標註對手領先之處。

| 對象 | 類型 | 斷網可用 | 可嵌入他人系統 | 瀏覽器零安裝 | 對 Nerilo 的意義 |
|---|---|---|---|---|---|
| Signal / WhatsApp | 主流加密 App | 否 | 否 | 否（需裝 App） | 不同戰場，不正面比 |
| Briar | 離線 mesh App（Android） | 是（藍牙／Tor） | 否 | 否 | 硬化 P2P 與資安稽核領先，是我們的敬畏對象；但綁 Android、非可嵌入層 |
| Berty / SimpleX | 去中心 App | 部分 | 否 | 否 | 技術扎實；仍是終端 App，不是可嵌入層 |
| Matrix / Element | 聯邦式協議 | 需伺服器 | 部分（重） | 是（需 homeserver） | 可相容而非對抗 |
| Nostr | 去中心 relay 協議 | relay 在才行 | 協議層可 | 是 | 可考慮相容；它是協議，Nerilo 是可嵌入實作與產品 |
| Meshtastic | LoRa 硬體 mesh | 是（需硬體） | 硬體 SDK | 否 | 不同層次；Nerilo 只當藍圖參考，不碰無線電頻段 |

結論：在「硬化 P2P、行動端成熟度、第三方資安稽核」上，Briar 等對手領先，我們不在他們主場比。Nerilo 佔的空位是右邊兩欄同時成立：可被嵌入，且瀏覽器零安裝。目前這一格是空的。

## 三道護城河，各自的真實狀態

一、定位牆（層而非 App）。這是最獨特、對手最難跟進的一點，因為跟進代價是放棄他們現有的 App 模式。狀態：已建成。可嵌入 SDK 的四道注入縫（signaling、名冊、儲存、身分）都可替換，核心靜態相依圖已與 Firebase 解耦，可發佈為套件。

二、觸及牆（瀏覽器原生、零安裝）。在災害當下、面對陌生人、沒有時間裝 App 的場景，這是決定性的。狀態：架構內建，是既有事實。

三、關係與敘事牆（開源可稽核、在地化、第一個真實使用者）。這道牆由信任資產構成，不由程式碼構成，也是對補助路線最關鍵的一道。狀態：部分。已有核心不變量稽核報告、25 篇架構決策紀錄、可查核的測試與 CI 閘門；缺的是第一個真實使用者與在地防災場景的落地驗證。

## 為什麼難抄，以及開源的悖論

必須誠實面對：Nerilo 開源，任何人能讀、能 fork，所以我們沒有程式碼層的護城河。加密與 P2P 是公開技術，抄得走。

因此牆必須在程式碼之外：

- 先行者位置。第一個把韌性通訊做成可嵌入 SDK 的實作，先建立 SDK 生態的初期網路效應。
- 執行速度。單人加 AI 的迭代速度，比委員會式的開源專案快，在特定利基上持續領先。
- 在地關係與信任。台灣災防場景的在地化、與防災單位的對接、補助帶來的機構信任，這是跨國團隊懶得跨、而非跨不過的地理與機構護城河。

抄襲者的正解不是把程式碼藏起來，那違背開源與補助精神；而是跑得比他快，並卡住他抄不走的東西：在地關係、機構信任、生態先行。

## 對誰、什麼場景

- 防災韌性。基礎設施受損時，社區內裝置互為節點，訊息仍能輾轉送達。
- 隱私與人權。通訊內容不落任何第三方之手，適合記者、社工、支援網絡等高風險溝通。
- 公共財。以 SDK 形式開放，讓影響力不限於單一 App，而是一層可被重複使用的基礎能力。

## 一頁總結

Nerilo 的護城河不在技術，在定位。把自己定義成別人拿去用的韌性通訊層，而非跟 Signal 搶用戶的 App。技術是拉開先行差距的引擎，不是牆本身。對補助路線而言，最該投資的是第三道牆：把開源可稽核、在地化、與第一個真實使用者做成別人抄不走的信任資產。
