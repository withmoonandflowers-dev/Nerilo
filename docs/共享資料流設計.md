# 共享資料流設計（區塊鏈式）

## 目標

在現有 P2P Mesh / Gossip 架構上，提供一種**類似區塊鏈的共享資料流**：

- **不可竄改的順序日誌**：每個節點維護同一份「只增不減」的日誌，並透過 hash 鏈驗證完整性。
- **多節點共享**：透過 Gossip 將新區塊/事件廣播給鄰居，最終全網對同一條鏈達成**最終一致**。
- **可驗證**：任一節點可驗證收到的區塊是否與前一區塊的 hash 銜接，以及 payload 的 hash 是否正確。

## 與區塊鏈的對應關係

| 區塊鏈概念 | 本設計對應 |
|-----------|------------|
| Block | **LedgerEntry**（一筆「帳本條目」） |
| Block hash | **entryHash**（SHA-256） |
| Previous block hash | **previousHash**（前一筆的 entryHash） |
| Merkle root / 交易 | **payloadHash**（payload 的 hash） |
| 共識（PoW/PoS） | **Gossip + 最終一致**（無挖礦，僅順序附加） |
| 全節點複製鏈 | **每個節點維護本地 `entries[]`，透過 Gossip 同步** |

## 資料結構

### LedgerEntry（帳本條目）

```
┌─────────────────────────────────────────────────────────┐
│ index         : number     // 條目序號（從 0 起）         │
│ previousHash  : string     // 前一筆的 entryHash          │
│ payloadHash   : string     // SHA256(JSON.stringify( payload )) │
│ timestamp     : number     // 建立時間                    │
│ creatorId     : string    // 建立者（如 mesh userId）     │
│ payload       : object    // 任意業務資料（事件內容）      │
│ signature?    : string    // 可選：建立者簽名             │
│ entryHash     : string    // 本筆的 hash（由上述欄位計算） │
└─────────────────────────────────────────────────────────┘
```

- **Hash 鏈**：`entryHash = SHA256(previousHash + index + timestamp + payloadHash + creatorId)`，形成「前一筆 → 本筆」的鏈式關係。
- **payloadHash**：保證 payload 未被竄改；驗證時重新計算 payload 的 hash 並與 `payloadHash` 比對。

### SharedStreamPayload（payload 類型）

可為任意 JSON 可序列化結構，例如：

- `{ type: 'chat', content: string, ... }`
- `{ type: 'room_event', action: 'join'|'leave', userId: string }`
- `{ type: 'sync', ... }`

由業務層定義；共享資料流只負責**順序、完整性與複製**，不解析業務語意。

## 資料流（共享鏈如何同步）

```
  Node A                Node B                Node C
    │                      │                     │
    │  append(payload)     │                     │
    │  → 計算 prevHash     │                     │
    │  → 建立 LedgerEntry  │                     │
    │  → 寫入本地 entries  │                     │
    │  → Gossip(entry) ────┼─────────────────────┼──→ 廣播給鄰居
    │                      │  handleEntry(entry) │
    │                      │  → 驗證 hash 鏈     │
    │                      │  → 驗證 payloadHash │
    │                      │  → append 到本地    │
    │                      │  → 再轉發 Gossip    │
    │                      │                     │  handleEntry(entry)
    │                      │                     │  → 同上
```

- **寫入**：只有「建立新條目」的節點會先算 `previousHash`（取自本地最後一筆的 `entryHash`），再計算本筆 `entryHash`，寫入本地並廣播。
- **接收**：收到 Gossip 的 LedgerEntry 時，先驗證 `entryHash` 與 `previousHash` 是否與本地最後一筆銜接，再驗證 `payloadHash`；通過則 append 到本地並可再轉發。
- **衝突**：若同一時刻多節點各自 append，可能出現分叉；本設計採用**最長鏈 + 先到先得**的簡單策略（或依 `index` + `timestamp` 決定接受哪一筆），不實作完整共識，僅達成「室內」級別的最終一致。

## 與現有 Mesh/Gossip 的整合方式

1. **承載方式**  
   - 方案 A：在現有 Gossip 訊息中增加 `type: 'ledger_entry'`，payload 為 `LedgerEntry`。  
   - 方案 B：獨立一個「Ledger 專用」Gossip 通道，只轉發 LedgerEntry。

2. **誰可以 append**  
   - 可限制為「房間內已註冊的 mesh 身分」；建立 LedgerEntry 時帶上 `creatorId`（如 mesh userId），驗證時可檢查是否在房間成員列表。

3. **訂閱與回放**  
   - UI 或服務層可訂閱 `SharedDataStream` 的 `onEntryAppended`；  
   - 新節點加入時可向鄰居請求「從 index 0 到最新」的條目列表以回放整條鏈（catch-up）。

## 使用情境

- **聊天室訊息順序**：每則訊息寫成一筆 LedgerEntry，全室看到相同順序與歷史。
- **房間事件日誌**：加入/離開、設定變更等寫入 stream，便於除錯與還原狀態。
- **輕量級審計軌跡**：不可竄改的順序記錄，適合小規模、信任模型較簡單的場景。

## 效能與安全措施

### 效能

- **Payload 大小**：單筆 payload 序列化後不得超過 `maxPayloadSize`（預設 100KB），避免過大 JSON 造成記憶體與 CPU 負擔。
- **Hash 輸入長度**：`sha256Hash` 單次輸入不得超過 1MB，避免 DoS。
- **條目數上限**：本地鏈條目數不得超過 `maxEntries`（預設 50,000），append 與 handleReceivedEntry 皆會檢查。
- **resetFromEntries**：一次最多接受 100,000 筆遠端條目，超過則拒絕。
- **seenEntryHashes**：超過 5,000 筆時清理一半，避免 Set 無限成長。

### 安全

- **Payload 型別**：僅接受純物件（`isPlainObject`），拒絕 `null`、陣列、`__proto__` 污染等。
- **Append 速率**：本地 append 每秒不得超過 `appendRateLimitPerSecond`（預設 20），防止濫用。
- **Entry 結構驗證**：`handleReceivedEntry` 與 `resetFromEntries` 會驗證 index、timestamp、creatorId、payload 為純物件，entryHash / payloadHash 為 64 字元 hex。
- **Hash 鏈驗證**：每筆 entry 的 previousHash、entryHash、payloadHash 皆須通過計算與格式檢查。

### 單元測試

- **crypto**：`sha256Hash`、`isPlainObject`、`isHex64`、`computePayloadHash`、`computeEntryHash` 的型別、長度、重複性與錯誤案例。
- **SharedDataStream**：append（含速率與 maxEntries）、handleReceivedEntry（合法、重複、鏈不符、index 不連續、結構不合法）、resetFromEntries、getEntries / getLastEntry、監聽與廣播。

## 使用範例

```typescript
import { SharedDataStream } from '@/core/mesh/SharedDataStream';

const stream = new SharedDataStream({
  roomId: 'room-1',
  creatorId: meshUserId,
  genesisPreviousHash: '0',
});

// 接到 Gossip：收到 LedgerEntry 時寫入本地
stream.setBroadcastHandler((entry) => {
  // 例如：meshGossipManager.broadcastLedgerEntry(entry);
  neighbors.forEach((n) => n.send({ type: 'ledger_entry', payload: entry }));
});

// 從 Gossip 收到別人廣播的 LedgerEntry 時
await stream.handleReceivedEntry(entry);

// 附加一筆新事件
await stream.append({ type: 'chat', content: 'hello' });

// 訂閱新條目
stream.onEntryAppended((entry) => {
  console.log('New entry', entry.index, entry.payload);
});
```

## 小結

本設計提供**類似區塊鏈的共享資料流**：hash 鏈、只增不減、多節點複製與驗證，並透過現有 Gossip 做傳輸與最終一致，不引入挖礦或複雜共識，適合室內/小群組的順序事件與審計需求。
