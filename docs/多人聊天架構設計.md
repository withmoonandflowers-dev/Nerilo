# 多人聊天架構設計

## 架構選擇

### 方案比較

1. **Mesh 拓撲（全連接）**
   - 每個用戶都與其他所有用戶建立 P2P 連線
   - 優點：無中心節點，延遲低
   - 缺點：連線數量 = n*(n-1)/2，當 n>5 時連線數過多，不穩定

2. **星型拓撲（中心節點）** ⭐ **推薦**
   - 房主作為中心節點，其他用戶都與房主建立連線
   - 優點：連線數量 = n-1，穩定且易於管理
   - 缺點：房主需要轉發訊息，但現代瀏覽器可以處理

3. **SFU（Selective Forwarding Unit）**
   - 需要額外的伺服器，成本高
   - 不適合純 P2P 架構

### 選擇：星型拓撲

**理由**：
- 穩定：連線數量線性增長
- 簡單：房主統一管理所有連線
- 易於擴展：可以支援 10+ 用戶
- 無需額外伺服器

## 架構設計

### 連線拓撲

```
        [房主/Initiator]
            /  |  \
           /   |   \
    [User1] [User2] [User3]
```

- 房主與每個用戶建立獨立的 P2P 連線
- 每個連線使用獨立的 DataChannel
- 房主負責轉發訊息給所有其他用戶

### 資料結構

```typescript
interface MultiP2PConnection {
  roomId: string;
  localUid: string; // 當前用戶 UID
  isHost: boolean; // 是否為房主
  connections: Map<string, P2PManager>; // remoteUid -> P2PManager
  messageRouter: MessageRouter; // 訊息路由器
}
```

### 訊息流程

#### 場景 1：非房主發送訊息

```
User1 -> [DataChannel] -> 房主 -> [轉發] -> User2, User3
```

#### 場景 2：房主發送訊息

```
房主 -> [DataChannel] -> User1, User2, User3 (並行)
```

## 實作方案

### 1. MultiP2PManager（多人 P2P 管理器）

**職責**：
- 管理多個 P2P 連線
- 處理新用戶加入/離開
- 轉發訊息給所有連線

**關鍵方法**：
- `addConnection(remoteUid: string)`: 建立與新用戶的連線
- `removeConnection(remoteUid: string)`: 移除連線
- `broadcastMessage(message: ChatMessage)`: 廣播訊息
- `forwardMessage(fromUid: string, message: ChatMessage)`: 轉發訊息（房主專用）

### 2. 連線建立流程

1. **新用戶加入房間**：
   - 更新 Firestore 參與者列表
   - 所有現有用戶收到訂閱更新

2. **房主建立連線**：
   - 房主檢測到新參與者
   - 建立新的 P2PManager 連線
   - 發送 offer 給新用戶

3. **新用戶接受連線**：
   - 新用戶收到 offer
   - 建立 P2PManager 連線
   - 發送 answer 給房主
   - 連線建立完成

4. **非房主用戶**：
   - 只與房主建立連線
   - 不與其他非房主用戶建立連線

### 3. 訊息路由

**房主邏輯**：
```typescript
// 收到訊息
onMessage(fromUid: string, message: ChatMessage) {
  // 如果是自己發的，直接廣播
  if (fromUid === this.localUid) {
    this.broadcastMessage(message);
  } else {
    // 轉發給其他所有用戶（除了發送者）
    this.forwardMessage(fromUid, message);
  }
}
```

**非房主邏輯**：
```typescript
// 收到訊息
onMessage(fromUid: string, message: ChatMessage) {
  // 直接顯示（因為所有訊息都來自房主）
  this.handleMessage(message);
}
```

## 限制與考量

### 連線數量限制

- **建議最大用戶數**：10-15 人
- **原因**：WebRTC 連線需要資源，過多連線可能導致效能問題

### 效能優化

1. **訊息去重**：
   - 使用 messageId 避免重複處理

2. **連線狀態管理**：
   - 監控每個連線的狀態
   - 自動重連失敗的連線

3. **訊息佇列**：
   - 如果連線未就緒，將訊息加入佇列
   - 連線建立後再發送
