# 完整測試流程

## 1. 程式與實際使用需求對照

### 1.1 需求摘要（來自實作交付總結與測試與驗收手冊）

| 需求類別 | 說明 | 程式對應 | 狀態 |
|---------|------|----------|------|
| 2 人聊天 | 星型拓撲、建立房間→等待→加入→聊天、雙向訊息 | ChatPage + P2PManager + RoomService | ✅ |
| 3+ 人聊天 | Mesh 架構、Gossip 傳訊、多人互傳 | MeshGossipManager + MeshChatService | ✅ |
| 架構自動選擇 | 2 人用星型、3+ 人用 Mesh | useP2PArchitecture.decide() | ✅ |
| 房間生命週期 | 建立、等待、open、關閉、建立新房間關閉舊房間 | RoomService + Firestore | ✅ |
| 等待房間 | 倒數、分享連結、參與者數、房主取消/非房主離開 | WaitingRoomPage | ✅ |
| 單人房間 | 單人可進等待、第二人加入後自動進聊天 | WaitingRoomPage 訂閱 + ChatPage join 後導向 | ✅ |
| 連線狀態 | idle → connecting → connected / failed / closed | ChatPage getConnectionStatusText() | ✅ |
| 訊息 | 發送、接收、空訊息不送、Enter 發送 | ChatPage + ChatService/MeshChatService | ✅ |
| 錯誤處理 | 不存在/已關閉房間導向 dashboard | ChatPage init + RoomService.getRoom | ✅ |
| Guest 測試環境 | 允許 guest 建立房間（VITE_ALLOW_GUEST_CREATE_ROOM） | playwright webServer env | ✅ |

### 1.2 結論

- **是否符合實際使用需求**：是。核心流程（建立房間、等待、加入、2 人/3+ 人聊天、連線狀態、訊息收發、房間關閉與錯誤導向）皆有實作並與需求對應。
- **已知限制**：Mesh 連線建立約 30–60 秒；E2E 需足夠超時（建議 60–120 秒）。

## 2. 完整測試流程（執行順序與依賴）

### 2.1 環境準備

1. 安裝依賴：`npm ci` 或 `npm install`
2. 確認 Firebase 測試/開發環境可連（或使用 Emulator）
3. 啟動 E2E 時由 Playwright 自動啟動 `npm run dev:test`（port 4173），無需手動起 server

### 2.2 建議執行順序（由快至慢、由穩至險）

| 順序 | 測試檔 | 說明 | 預估時間 |
|-----|--------|------|----------|
| 1 | waiting-room.spec.ts | 等待頁：建立、倒數、分享、第二人轉聊天、參與者數、取消/離開 | ~1–2 min |
| 2 | room-timeout.spec.ts | 超時：倒數與超時 UI 存在 | ~30 s |
| 3 | room-closed.spec.ts | 關閉：訪問已關閉、建新關舊、兩人加入後不關閉 | ~1–2 min |
| 4 | single-user-room.spec.ts | 單人：建立→等待、第二人加入後連線與訊息 | ~1 min |
| 5 | user-chat.spec.ts | 2 人：建立、連線、雙向傳訊 | ~1 min |
| 6 | room-management.spec.ts | 房間管理：關舊房、第二人轉聊天、多條訊息 | ~1–2 min |
| 7 | guest-chat.spec.ts | Guest：兩 guest 建立房間並傳訊 | ~1 min |
| 8 | comprehensive-chat.spec.ts | 完整：權限、生命週期、連線狀態、訊息、架構、錯誤、效能 | ~3–5 min |
| 9 | architecture-selection.spec.ts | 架構：2 人星型、3 人 Mesh、2→3 人 | ~3–5 min |
| 10 | mesh-gossip.spec.ts | Mesh：3 人/5 人 Gossip 傳訊（需較長超時） | ~3–6 min |

### 2.3 單一指令執行全部

```bash
npm run test:e2e
```

### 2.4 分組指令（供腳本使用）

- **快速回歸（不含 Mesh）**  
  `npm run test:e2e -- tests/e2e/waiting-room.spec.ts tests/e2e/room-management.spec.ts tests/e2e/user-chat.spec.ts tests/e2e/single-user-room.spec.ts tests/e2e/room-closed.spec.ts tests/e2e/room-timeout.spec.ts tests/e2e/guest-chat.spec.ts`

- **完整功能（含 comprehensive）**  
  `npm run test:e2e -- tests/e2e/comprehensive-chat.spec.ts`

- **架構與 Mesh（長超時）**  
  `npm run test:e2e -- tests/e2e/architecture-selection.spec.ts tests/e2e/mesh-gossip.spec.ts --timeout=120000`

## 3. 流程完整性檢討

### 3.1 已涵蓋

- 認證/權限（Guest 導向、建立房間按鈕）
- 房間生命週期（建立→等待→加入→聊天→離開、建新關舊）
- 等待房間（倒數、分享、參與者、取消/離開、第二人轉聊天）
- 單人房間（第二人加入後自動進聊天與連線）
- 2 人連線與多條訊息
- 3 人/5 人 Mesh Gossip
- 架構選擇（2 人星型、3 人 Mesh）
- 錯誤處理（不存在/已關閉房間→dashboard）
- 連線狀態、空訊息、Enter 發送、快速多訊息

### 3.2 未涵蓋或部分涵蓋

- **影音/檔案**：測試與驗收手冊有 P2P 影音、檔案傳送；目前 E2E 僅文字聊天，未自動化影音/檔案。
- **跨裝置/離線**：手冊有同帳號兩裝置、離線不補同步；E2E 未涵蓋。
- **Firestore 稽核**：手冊要求確認無聊天/檔案/影音寫入 Firestore；需手動或另寫測試。
- **真實超時**：room-timeout 僅檢查倒數與超時 UI 存在，未實際等 5 分鐘觸發超時。

### 3.3 結論

- 以「文字聊天 + 房間 + 架構選擇」為範圍，**流程完整**。
- 若要對齊完整驗收手冊，需補：影音/檔案 E2E、Firestore 稽核、可選短超時參數的 room-timeout。

## 4. 程式與測試流程對照（Review）

### 4.1 UI 與選擇器

| 測試使用 | 程式位置 | 符合 |
|----------|----------|------|
| `.role-badge`、guest 文字 | DashboardPage | ✅ |
| 「+ 建立新房間」「建立房間」 | DashboardPage | ✅ |
| `/waiting/:id`、等待連線、剩餘時間、`.timer-value` | WaitingRoomPage | ✅ |
| 分享房間連結、複製連結、目前參與者、取消房間/離開 | WaitingRoomPage | ✅ |
| `/chat/:id`、聊天室:、已連線、輸入訊息、傳送 | ChatPage | ✅ |
| `.connection-status`、`.message` | ChatPage + ChatPage.css | ✅ |
| 「← 返回」按鈕（/返回|離開/i） | ChatPage | ✅ |

### 4.2 行為對應

- 第二人加入後轉聊天：WaitingRoomPage 訂閱 `status === 'open'` → `navigate(/chat/:id)`；ChatPage join 後若 `status === 'open'` 或 participants >= 2 會進聊天。✅
- 建立新房間關閉舊房間：RoomService 在 createRoom 時關閉該用戶其他房間。✅
- 已關閉/不存在房間導向 dashboard：ChatPage init 中 getRoom/status 檢查後 `navigate('/dashboard')`。✅
- 2 人星型、3+ 人 Mesh：useP2PArchitecture.decide() 與 ChatPage initializeP2P 一致。✅

### 4.3 結論

- 目前 E2E 使用的選擇器與流程與程式行為一致，**程式符合既有測試流程**。若測試失敗，多為時序/網路（連線或 Firestore 延遲）或超時不足，而非需求與實作不一致。

## 5. 修正程式直到通過測試（本文件對應執行結果）

- 測試執行結果與任何失敗案例，應記錄於 **測試執行報告**（或 CI 產出）。
- 修正原則：
  - 失敗若為**超時**：在 playwright.config 或該 test 的 `timeout` 調高（Mesh 建議 90–120s）。
  - 失敗若為**選擇器**：對齊實際 DOM（如按鈕文字、class）。
  - 失敗若為**流程**：對照本文件 §2、§4 調整程式或測試步驟順序/等待條件。
- 重跑對應套件直到通過後，再跑完整 `npm run test:e2e` 做最終確認。

### 5.1 依賴 WebRTC 的測試（可能不穩定）

以下測試依賴 **WebRTC P2P 連線** 與 **Firebase/Firestore** 即時同步，在部分環境（如 CI、網路限制、Firebase 延遲）下「已連線」可能超過 45 秒才出現或無法建立：

- `user-chat.spec.ts`：兩個已登入使用者建立房間、連線並互相傳訊息
- `room-management.spec.ts`：兩個使用者可以成功連線並互相發送多條訊息

**已做程式修正**：ChatPage 與 useRoomSubscription 在房間狀態為 `open` 時，若讀到參與者數 0 或 1，會視為 Firestore 同步延遲並以至少 2 人初始化 P2P，避免因快取導致不初始化。

**若仍失敗**：可依環境選擇（1）提高「已連線」等待時間（例如 60–90s）、（2）在 CI 中僅跑不依賴連線的 E2E、（3）本地手動驗證 2 人聊天後再交付。

## 6. 執行腳本需求與腳本位置

### 6.1 是否需要執行腳本

- **需要**。理由：
  - 一次跑完全部 E2E 耗時長；需「快速回歸」與「完整/含 Mesh」分開。
  - CI 或本機需可重現的指令（依賴安裝、可選 port、超時）。
  - 新成員需單一入口執行測試與查看報告。

### 6.2 腳本位置與用途

- 腳本放在專案 **`scripts/`** 目錄下（Windows PowerShell）：
  - **`scripts/test-e2e-quick.ps1`**：E2E 快速回歸（不含 Mesh）
  - **`scripts/test-e2e-full.ps1`**：E2E 完整套件（comprehensive、architecture、mesh）
  - **`scripts/test-e2e-all.ps1`**：全部 E2E
  - **`scripts/run-all-tests.ps1`**：單元測試 + 全部 E2E 一鍵
- 說明與用法見 **`scripts/README.md`**。
- 後續可參考該 README 與上述腳本擴充（例如只跑某個 spec、或加環境變數）。
