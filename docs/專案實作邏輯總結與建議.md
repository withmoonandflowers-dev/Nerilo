# 專案實作邏輯總結與建議

## 📐 專案架構概述

### 技術棧
- **前端**：React 18 + TypeScript + Vite
- **狀態管理**：React Context (AuthContext, FeatureContext)
- **路由**：React Router v6
- **後端**：Firebase (Auth, Firestore, Functions, Hosting)
- **P2P 通訊**：WebRTC (RTCPeerConnection, RTCDataChannel)
- **測試**：Playwright (E2E)
- **樣式**：CSS Modules

### 核心模組

#### 1. 認證與權限系統
```
AuthContext
  ├─ Firebase Auth (Email/Password, Google)
  ├─ Custom Claims (guest, user, admin)
  └─ 角色管理

FeatureContext
  ├─ 功能註冊 (Firestore features 集合)
  ├─ 角色過濾 (requiredRoles)
  └─ 動態功能列表
```

#### 2. 房間管理系統
```
RoomService
  ├─ createRoom() - 建立房間（自動清理舊房間）
  ├─ joinRoom() - 加入房間（自動激活）
  ├─ leaveRoom() - 離開房間
  ├─ closeRoom() - 關閉房間
  ├─ updateMeshIdentity() - 更新 Mesh 身分
  └─ subscribeRoom() - 監聽房間變化
```

#### 3. P2P 通訊架構

##### 星型拓撲（2 人）
```
P2PManager
  ├─ P2PConnectionManager (WebRTC 連線)
  ├─ P2PChannelBus (DataChannel 管理)
  └─ ChatService (訊息處理)
```

##### Mesh 架構（3+ 人）
```
MeshGossipManager
  ├─ IdentityManager (密鑰管理)
  ├─ SecurityManager (簽名驗證)
  ├─ MeshTopologyManager (拓撲管理)
  ├─ GossipMessageHandler (Gossip 協議)
  └─ MeshChatService (聊天服務)
```


### 2. 建立房間流程

```typescript
// 步驟詳解
1. 用戶點擊「建立新房間」
   ↓
2. 權限檢查（前端）
   - 檢查 user.role !== 'guest'
   - 檢查 user.uid 存在
   ↓
3. 關閉舊房間（後端）
   - RoomService.closeAllUserRooms(ownerUid)
   - 查詢該用戶的所有房間
   - 將所有房間狀態設為 'closed'
   ↓
4. 建立新房間
   - 生成 roomId (UUID)
   - 寫入 Firestore p2pRooms/{roomId}
   - 狀態設為 'waiting'
   ↓
5. 導向等待頁面
   - navigate(`/waiting/${roomId}`)
```

### 3. 加入房間流程

```typescript
// 步驟詳解
1. 用戶訪問房間 URL (/chat/{roomId})
   ↓
2. 加入房間（後端）
   - RoomService.joinRoom(roomId, uid)
   - 檢查房間是否存在
   - 檢查是否已加入（避免重複）
   - 添加到 participants 陣列
   ↓
3. 檢查房間狀態
   - 如果 status === 'waiting' && participants.length >= 2
   - 自動將狀態改為 'open'
   ↓
4. 監聽房間變化
   - RoomService.subscribeRoom(roomId, callback)
   - 當房間狀態或參與者變化時觸發
   ↓
5. 根據狀態導航
   - status === 'closed' → navigate('/dashboard')
   - status === 'waiting' → navigate(`/waiting/${roomId}`)
   - status === 'open' → 初始化 P2P
```

### 4. P2P 初始化流程

```typescript
// 架構選擇
const useMesh = room.topology === 'mesh' || participants.length >= 3;

if (useMesh) {
  // Mesh 架構初始化
  1. 建立 MeshChatService
  2. 初始化 MeshGossipManager
     - IdentityManager.initialize() (生成/載入密鑰)
     - 註冊身分到 Firestore
     - MeshTopologyManager.initialize() (發現節點、選擇鄰居)
     - GossipMessageHandler.initialize()
  3. 建立鄰居連線（非同步，k=6）
  4. 等待連線建立（30-60 秒）
} else {
  // 星型拓撲初始化
  1. 建立 P2PManager
  2. 初始化 P2PConnectionManager
     - 建立 RTCPeerConnection
     - 設置 ICE 伺服器
     - 監聽 Firestore signals
  3. 如果是 initiator，建立 DataChannel 並發送 offer
  4. 如果是 non-initiator，等待遠端 DataChannel
  5. 等待 ChannelBus 準備（狀態變為 'open'）
  6. 建立 ChatService
}
```

### 5. 訊息發送流程

#### 星型拓撲
```typescript
1. 用戶輸入訊息並點擊「傳送」
   ↓
2. ChatService.sendMessage()
   - 建立 ChatMessage 物件
   - 儲存到 IndexedDB
   ↓
3. 封裝為 P2PEnvelope
   - ns: 'chat'
   - type: 'MSG_SEND'
   - payload: ChatMessage
   ↓
4. ChannelBus.send(envelope)
   - 透過 RTCDataChannel 發送
   ↓
5. 如果是房主，轉發給其他用戶
   - MultiP2PManager.forwardMessage()
```

#### Mesh 架構
```typescript
1. 用戶輸入訊息並點擊「傳送」
   ↓
2. GossipMessageHandler.sendMessage()
   - Rate limiting 檢查
   - 序列號遞增 (seq++)
   - 建立 GossipMessage（不含簽名）
   ↓
3. 簽名訊息
   - SecurityManager.signMessage()
   - 使用 ECDSA P-256 私鑰簽名
   ↓
4. 隨機選 2 個鄰居發送
   - MeshTopologyManager.getNeighbors()
   - selectRandomNeighbors(neighbors, 2)
   - MeshConnection.send(message)
   ↓
5. 鄰居接收並轉發
   - 驗證簽名
   - 檢查序列號（防重放）
   - 檢查 TTL
   - 顯示訊息
   - 如果 TTL > 0，轉發給其他 2 個鄰居
```

## 🎯 架構設計決策

### 1. 為什麼選擇雙架構？

**星型拓撲（2 人）**：
- ✅ 簡單直接，連線建立快速
- ✅ 低延遲，訊息直接傳輸
- ✅ 資源消耗少

**Mesh 架構（3+ 人）**：
- ✅ 可擴展性（支援多人）
- ✅ 容錯性（部分節點斷線仍可運作）
- ✅ 安全性（訊息簽名驗證）

### 2. 為什麼使用自動選擇？

- **用戶體驗**：無需手動配置，系統自動選擇最適合的架構
- **向後兼容**：2 人聊天保持原有體驗
- **未來擴展**：可以強制使用特定架構（topology 欄位）

### 3. 為什麼 Mesh 需要較長時間？

- **身分註冊**：每個節點需要註冊到 Firestore（1-2 秒）
- **節點發現**：等待其他節點註冊並同步（2-5 秒）
- **連線建立**：每個節點需要建立 k=6 個連線（10-30 秒）
- **WebRTC Signaling**：每個連線需要完成 offer/answer/ICE（5-15 秒）

**總計**：30-60 秒是正常的，這是 Mesh 架構的特性。

## 📊 資料流圖

### Firestore 資料結構

```
p2pRooms/{roomId}
  ├─ ownerUid: string
  ├─ participants: string[]
  ├─ status: 'waiting' | 'open' | 'closed'
  ├─ meshIdentities?: {
  │     [firebaseUid]: {
  │       userId: string (hash(pubKey))
  │       pubKey: string (Base64)
  │       joinedAt: number
  │     }
  │   }
  ├─ topology?: 'star' | 'mesh'
  └─ signals/{signalId}
      ├─ from: string
      ├─ to?: string
      ├─ type: 'offer' | 'answer' | 'ice'
      └─ payload: any
```

### 訊息儲存

- **Firestore**：不儲存聊天內容（僅 signaling 和 metadata）
- **IndexedDB**：本地儲存聊天訊息（清除瀏覽器資料即永久遺失）
- **記憶體**：當前會話的訊息（用於 UI 顯示）

## 🔐 安全性實作

### 1. 三層權限驗證

```
前端檢查 (DashboardPage)
    │
    ▼ (通過)
後端檢查 (RoomService)
    │
    ▼ (通過)
Firestore 規則 (firestore.rules)
    │
    ▼ (通過)
允許操作
```

### 2. 訊息安全（Mesh）

- **簽名驗證**：每條訊息都使用 ECDSA P-256 簽名
- **防重放**：使用序列號檢查
- **防篡改**：簽名包含所有訊息欄位
- **去重**：使用 messageId（hash）去重

### 3. 速率限制

- **發送速率**：每秒最多 10 條訊息
- **訊息去重**：最多儲存 10,000 個已見過的訊息 ID
- **序列號檢查**：允許最多 100 的序列號間隙

### 1. 訊息批次處理

```typescript
// 當前：每條訊息單獨處理
// 建議：批次處理多條訊息
class MessageBatcher {
  private batch: ChatMessage[] = [];
  private batchSize = 10;
  
  add(message: ChatMessage) {
    this.batch.push(message);
    if (this.batch.length >= this.batchSize) {
      this.flush();
    }
  }
  
  private async flush() {
    await indexedDBService.saveChatMessages(this.batch, roomId);
    this.batch = [];
  }
}
```
