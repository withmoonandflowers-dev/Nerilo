# Nerilo 系統架構文件

## 1. 概述

Nerilo 是一個基於 Firebase + WebRTC 的 P2P 即時互動平台，提供文字聊天、音訊/視訊通話、檔案傳送等功能。所有使用者資料（聊天文字、檔案、影音）僅透過 P2P 傳輸，不儲存於伺服器端。

## 2. 系統架構

### 2.1 整體架構

系統採用前後端分離架構：

- **前端**：React + TypeScript + Vite
- **後端**：Firebase (Auth, Firestore, Functions, Hosting)
- **P2P 通訊**：WebRTC (RTCPeerConnection, DataChannel, MediaStream)

### 2.2 核心元件

#### 2.2.1 共用 P2P 通訊層

**P2PConnectionManager**
- 管理 RTCPeerConnection 生命週期
- 處理 Firestore signaling (offer/answer/ICE)
- 注入 ICE servers（可選：從 Cloud Functions 取得 Twilio TURN）
- 監聽連線狀態變化

**P2PChannelBus**
- 管理 RTCDataChannel
- 提供 `send()` / `subscribe()` API
- 實作序列化、流控（bufferedAmount）、錯誤處理
- 支援多路復用（讓多功能共用同一 DataChannel）

**P2PProtocolRegistry**
- 提供 namespace/type/payload schema 註冊機制
- 驗證 Envelope 基本欄位與 payload 必填欄位
- 分派消息給對應功能 handler
- 遇到未知 ns/type 回傳 system.ERROR

**P2PFileTransferService**
- 實作 file 協議（FILE_META/FILE_CHUNK/FILE_END/FILE_CANCEL）
- 分片策略、重組、進度事件
- 支援取消傳輸

**P2PMediaService**
- getUserMedia、track 管理
- 狀態同步（透過 media namespace）
- 支援 mute/unmute、camera on/off

#### 2.2.2 功能模組

**ChatService**
- 處理文字訊息收發
- 支援編輯、刪除訊息
- 輸入中狀態（TYPING）
- 與 IndexedDB 整合儲存歷史紀錄

**SyncService**
- 同帳號跨裝置同步
- 索引比對、差異同步
- 衝突解決策略（時間戳優先）

#### 2.2.3 服務層

**IndexedDBService**
- 使用 Dexie 管理 IndexedDB
- 儲存聊天紀錄、檔案 metadata、系統事件
- 提供查詢、更新、刪除 API

## 3. 資料流

### 3.1 連線建立流程

1. 使用者 A 建立房間 → 寫入 Firestore `p2pRooms/{roomId}`
2. 使用者 A 初始化 P2PConnectionManager → 建立 RTCPeerConnection
3. 使用者 A 建立 DataChannel → 建立 offer → 寫入 Firestore `signals`
4. 使用者 B 加入房間 → 讀取 Firestore `p2pRooms/{roomId}`
5. 使用者 B 監聽 `signals` → 收到 offer → 建立 answer → 寫入 Firestore
6. 雙方交換 ICE candidates → 建立 P2P 連線

### 3.2 資料傳輸流程

**文字訊息**
1. 使用者輸入 → ChatService.sendMessage()
2. 封裝為 Envelope (ns: "chat", type: "MSG_SEND")
3. P2PChannelBus.send() → RTCDataChannel
4. 接收端解析 Envelope → 儲存 IndexedDB → 更新 UI

**檔案傳送**
1. 選擇檔案 → P2PFileTransferService.sendFile()
2. 發送 FILE_META → 分片傳送 FILE_CHUNK → 發送 FILE_END
3. 接收端重組檔案 → 儲存 IndexedDB → 提供下載

**影音通話**
1. getUserMedia() → P2PMediaService.startLocalMedia()
2. addTrack() → RTCPeerConnection
3. 遠端 ontrack 事件 → 顯示遠端影像
4. 狀態變更透過 MEDIA_TOGGLE 同步

### 3.3 資料隱私保護

**硬性原則**：
- 聊天文字、檔案本體、影音內容 **不得** 寫入 Firestore
- Firestore 僅用於 signaling、功能註冊、使用者 profile、房間 metadata
- 所有使用者資料僅透過 P2P 傳輸
- 本機儲存僅存於 IndexedDB（清除瀏覽器資料即永久遺失）

## 4. 權限控制 (RBAC)

### 4.1 角色定義

- **guest**：未登入使用者
- **user**：一般登入者
- **admin**：管理者

### 4.2 權限來源

- Firebase Auth Custom Claims 為權威來源
- Firestore Security Rules 作為最終存取裁決者

### 4.3 權限規則

- P2P 功能僅 user/admin 可使用
- guest 不可建立/加入 P2P 房間
- features 依 requiredRoles 過濾顯示

## 5. 協議設計

### 5.1 統一 Envelope 格式

所有非檔案二進位資料必須符合：

```typescript
{
  v: 1,
  ns: "chat|file|media|sync|system|feature.xxx",
  type: "STRING",
  id: "UUID",
  ts: 1710000000000,
  from: "uid/deviceId",
  to?: "uid/deviceId",
  replyTo?: "UUID",
  payload: {},
  meta?: {}
}
```

### 5.2 官方保留 Namespace

- **system**：ERROR, PING, PONG, ACK
- **chat**：MSG_SEND, MSG_EDIT, MSG_DELETE, TYPING
- **file**：FILE_META, FILE_CHUNK, FILE_END, FILE_CANCEL
- **media**：MEDIA_READY, MEDIA_TOGGLE, MEDIA_END
- **sync**：INDEX, REQ, RES, DONE

### 5.3 功能自訂 Namespace

格式：`feature.<name>`（例如：`feature.crm`）

必須透過 P2PProtocolRegistry 註冊才能使用。

## 6. 擴充性設計

### 6.1 新功能接入流程

1. 在 Firestore `features/{featureId}` 註冊功能
2. 定義 namespace 與 types
3. 實作功能 handler
4. 透過 P2PProtocolRegistry.register() 註冊
5. 前端新增 route 與 UI

**不需修改**：
- P2P 核心程式碼
- Menu 核心程式（動態載入）

## 7. 安全性

### 7.1 傳輸層加密

- WebRTC 內建 DTLS/DTLS-SRTP 加密
- 不要求應用層 E2EE（可選擴充）

### 7.2 Twilio TURN 安全

- Twilio credential 不可存在前端
- 必須由 Cloud Functions 取得短效 iceServers
- 前端僅接收短效 token

### 7.3 Firestore Security Rules

- 明確禁止儲存聊天/檔案/影音內容
- 僅 participants 可讀寫 signaling
- signals 為 append-only（不可更新/刪除）

## 8. 效能考量

### 8.1 DataChannel 流控

- 監控 bufferedAmount
- 設定 bufferedAmountLowThreshold
- 實作發送佇列避免 UI 卡死

### 8.2 檔案傳輸

- 分片大小：64KB（可調整）
- 非同步處理避免阻塞
- 進度回調機制

### 8.3 IndexedDB 查詢

- 使用索引加速查詢
- 限制查詢數量（預設 100 筆）
- 反向排序顯示最新訊息
