# 模組化與可插拔架構

本文件說明專案的模組化策略、依賴反轉（Ports & Adapters）與功能可插拔設計，以利解耦、可測試與可擴充。

---

## 1. 設計原則

- **依賴反轉**：功能模組依賴「介面（Ports）」，不直接依賴具體實作（Firestore、IndexedDB）。實作由應用組裝時注入。
- **可插拔功能**：每個功能（如聊天）以「註冊」方式提供路由與 UI，App 僅負責組裝路由與提供服務。
- **可測試性**：測試時可注入 Mock 實作（如 MockRoomService、MockChatStorage），無需啟動 Firebase。

---

## 2. 分層與職責

| 層級 | 目錄 | 職責 |
|------|------|------|
| **Ports** | `src/ports/` | 介面定義（IRoomService、IChatStorage），不依賴框架與具體儲存 |
| **Adapters** | `src/services/`、`src/config/` | 實作 Ports（如 RoomService、IndexedDBService），依賴 Firebase/Dexie |
| **Core** | `src/core/` | P2P/Mesh 通訊邏輯，依賴 Ports 或具體傳輸介面 |
| **Features** | `src/features/<name>/` | 功能模組：UI + 業務邏輯，透過 Context 取得服務，對外僅暴露路由與元件 |
| **App** | `src/App.tsx`、`src/main.tsx` | 組裝：Providers、功能註冊表、路由 |

---

## 3. Ports（介面）

### 3.1 IRoomService

房間生命週期與訂閱，供「等待房、聊天房、Dashboard」使用。

- `createRoom(...)`、`getRoom(roomId, forceServer?)`、`subscribeRoom(roomId, callback)`、`joinRoom`、`leaveRoom`、`closeRoom`、`closeAllUserRooms`、`subscribeUserRooms`、`subscribePublicRooms`、`activateRoom`、`updateMeshIdentity`、`getMeshIdentities`

### 3.2 IChatStorage

聊天訊息持久化（目前為 IndexedDB），供 ChatService/MeshChatService 寫入與讀取。

- `saveChatMessage(message, roomId)`、`getChatMessages(roomId, limit?)`

功能若僅需「送/收即時訊息」可只依賴 P2P/Mesh；若需「離線紀錄」則依賴 IChatStorage。

---

## 4. 服務注入（ServicesContext）

- **ServicesContext** 提供 `roomService: IRoomService`、`chatStorage: IChatStorage`。
- 預設值為目前實作（RoomService 的適配實例、IndexedDBService 實例）。
- 測試或 Storybook 可包一層 `ServicesProvider` 並傳入 Mock，達成可插拔與可測試。

---

## 5. 功能註冊表（可插拔）

- **Route 註冊**：各功能在 `src/features/<name>/index.ts` 導出 `route: { path, component }`。
- **App** 從 `features/registry` 讀取註冊表，動態渲染 `<Route path={...} element={...} />`。
- 靜態路由（登入、Dashboard、等待房）仍寫在 App，因與認證/流程強相關。

---

## 6. 模組邊界

- **features/chat**：對外只暴露 `ChatPage` 與 `route`；內部 hooks、ChatService、MeshChatService 不從外部直接引用。
- **core/p2p、core/mesh**：可被多個功能共用，不依賴 React；若需「房間」僅依賴 IRoomService 或透過參數傳入。
- **services/**：實作 Ports，可被替換為 Mock 或其它後端（如本地 SDK）。

---

## 7. 測試策略

- **單元測試**：對依賴 IRoomService / IChatStorage 的邏輯，注入 Mock（回傳固定 room、固定訊息列表）。
- **E2E**：使用真實 Firebase 與服務，驗證完整流程。
- **整合測試**：可選：用 MemoryAdapter 等假後端，不連線 Firebase。

此架構使功能模組化、解耦且可插拔，並方便針對業務邏輯撰寫單元測試。

---

## 8. 依賴規則（強制）

- **Features / Pages**：不得直接 `import` `RoomService`、`IndexedDBService` 或 `config/firebase`；僅能透過 `useServices()` 取得 Port 實例。
- **新功能**：必須在 `src/features/registry.tsx` 註冊路由。
- **Ports**：僅定義介面，不依賴框架或具體儲存。
- **Adapters**：實作 Ports，由 App 或 Context 組裝與注入。

詳見 **`docs/開發規範.md`**。

---

## 9. 運行時可驗證日誌（強制）

為確保每個功能在實際運行時有被執行，E2E 可透過 console 斷言驗證：

- 使用 **`featureLog(feature, action, payload?)`**（`src/utils/featureLog.ts`）。
- 輸出格式：`[NERILO:feature:action]`，例如 `[NERILO:chat:message_sent]`。
- 每個功能在關鍵路徑（進入、送出、離開等）至少呼叫一次 `featureLog`。
- E2E 測試應在關鍵流程後收集 console，並斷言出現對應的 `NERILO:feature:action`。

詳見 **`docs/開發規範.md`** § 2。

---

## 10. 測試範例（Mock 注入）

單元測試可注入 Mock 實作，無需連線 Firebase：

```ts
const mockRoomService: IRoomService = {
  getRoom: async () => ({ roomId: 'test-1', status: 'open', ... }),
  subscribeRoom: () => () => {},
  // ... 其他方法回傳預設值
};
// 在測試元件外包 <ServicesProvider value={{ roomService: mockRoomService }} />
```

專案內範例：`tests/unit/ServicesContext.spec.ts`。

---

## 11. 功能契約（摘要）

- 每個 **feature** 對外僅暴露：註冊在 `registry` 的 route（path + element）。
- 功能內部取得能力僅透過：**Context**（`useServices`、`useAuth`、`useFeatures`）或 **props**。
- 每個功能必須在關鍵路徑使用 **featureLog**，供 E2E 驗證實際有執行。
