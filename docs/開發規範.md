# 開發規範（強制遵守）

本規範適用於所有人與 AI 協作時的程式改動，務必符合以下條款，以維持架構一致與可驗證性。

---

## 1. 架構與依賴（強制）

### 1.1 依賴方向

- **Features（`src/features/*`）**  
  - 僅能透過 **`useServices()`** 取得 `roomService`、`chatStorage`。  
  - **禁止** 直接 `import` `RoomService`、`IndexedDBService` 或 `config/firebase`。  
  - 允許：`import` from `ports`、`contexts`、`types`、`utils`、`core`、同 feature 內檔案。

- **Pages（`src/pages/*`）**  
  - 同上：房間/聊天相關邏輯透過 `useServices()`，不直接依賴具體 Adapter。

- **Ports（`src/ports/*`）**  
  - 僅定義介面，不依賴 React、Firebase、Dexie。

- **Adapters（`src/services/*`、`src/config/*`）**  
  - 實作 Ports 或提供給 App 組裝的實例；可被 Context 注入替換（如測試用 Mock）。

### 1.2 功能註冊

- 新增「可路由」功能時，必須在 **`src/features/registry.tsx`** 註冊 `path` 與 `element`。  
- 靜態路由（登入、Dashboard、等待房）維持在 `App.tsx`。

### 1.3 禁止項目

- 在 features 或 pages 中直接 `import { RoomService } from '.../RoomService'` 或 `import { indexedDBService } from '.../IndexedDBService'`。  
- 在 features 中 `import { db, auth } from '.../config/firebase'`（改由 Context 或 Port 取得能力）。

---

## 2. 運行時可驗證日誌（強制）

### 2.1 目的

確保每個功能在實際運行時有被執行，E2E 可透過 console 斷言驗證。

### 2.2 使用方式

- 使用 **`featureLog(feature, action, payload?)`**（`src/utils/featureLog.ts`）。  
- 輸出格式：`[NERILO:feature:action]` 或 `[NERILO:feature:action] { ... }`。  
- 每個功能在 **關鍵路徑** 至少呼叫一次可識別的 `featureLog`（例如：進入頁面、送出訊息、建立房間、離開房間）。

### 2.3 建議 action 對照

| 功能   | 建議 action                |
|--------|----------------------------|
| chat   | init, room_joined, architecture_decided, message_sent, leave_room |
| dashboard | room_created, join_room_clicked, room_joined_from_list |
| waiting  | room_subscribed, room_open_redirect, cancel_or_leave, activate_room |
| auth   | login, logout              |

### 2.4 E2E 驗證

- E2E 測試應在關鍵流程後，收集頁面 console，並 **assert** 出現對應的 `NERILO:feature:action` 字串。  
- 範例：guest-chat 流程結束後，斷言 logs 中曾出現 `NERILO:chat:init`、`NERILO:chat:message_sent` 等。

---

## 3. 變更後必做（與 AI 協作規範一致）

- 執行 `npm run type-check`、`npm run lint`、`npm run test:run`。  
- 若改動影響房間/聊天/等待頁/Firestore，需跑相關 E2E 或手動驗證。  
- 新增或修改功能時，確認已加入對應的 `featureLog`，且 E2E 能覆蓋該路徑並可選地斷言日誌。
