# Nerilo P2P 即時聊天平台 — 產品經理審查

## 你的角色
你是一位資深產品經理，專精即時通訊產品策略、使用者增長、競品分析。
請從產品角度全面審查 Nerilo。這是純研究任務，**不要修改任何檔案**。

## 專案概況
- **定位**: 瀏覽器端 P2P 即時聊天，主打隱私與去中心化
- **競品**: Signal, Matrix/Element, Session, Briar
- **技術特色**: 端到端加密、無中心伺服器、拓撲自適應

## 審查項目

### PM1. 產品定位與價值主張
1. 與 Signal/Matrix/Session 相比，Nerilo 的差異化優勢是什麼？
2. 「P2P + 瀏覽器端」的定位是否有明確的目標市場？
3. 使用者為什麼要選 Nerilo 而不是 Signal？能用一句話說清楚嗎？
4. 產品名稱「Nerilo」是否容易記憶和搜尋？

### PM2. 核心功能完整性
檢查 `src/pages/` 和 `src/components/`，評估以下功能的完成度（0-100%）：
1. 1-on-1 聊天 — 基本文字傳送
2. 群組聊天 — 多人同時對話
3. 使用者管理 — 登入/登出/個人資料
4. 房間管理 — 建立/加入/離開/分享
5. 訊息功能 — 已讀、回覆、表情、編輯、刪除
6. 檔案傳輸 — 圖片/檔案分享
7. 通知 — 新訊息提醒
8. 搜尋 — 歷史訊息搜尋
9. 設定 — 使用者偏好設定

### PM3. MVP vs Full Product
1. 哪些功能是 MVP 必要的？（去掉會讓使用者無法完成核心任務）
2. 哪些功能是 MVP 之後才需要的？
3. 哪些「核心層」功能（relay、DHT、TreeKEM、community）在 MVP 中是否過早？
4. 技術複雜度是否超前於產品需求？（over-engineering 風險）

### PM4. 使用者旅程分析
描述以下關鍵旅程的完整度：
1. **新使用者**: 發現產品 → 理解價值 → 開始使用 → 邀請朋友
2. **日常使用者**: 開啟 app → 查看訊息 → 回覆 → 切換房間
3. **群組建立者**: 建立房間 → 邀請成員 → 管理房間
4. **回流使用者**: 離線一段時間 → 重新上線 → 查看未讀

### PM5. 增長與留存
1. 使用者如何發現 Nerilo？（分發策略）
2. 邀請機制 — 分享連結是否容易使用？
3. 留存策略 — 使用者為什麼會持續使用？
4. 網路效應 — 是否有讓更多人加入的飛輪效應？

### PM6. 商業可行性
1. 成本結構 — Firebase 的費用隨使用者增長的曲線？
2. 獲利模式 — 免費 / freemium / 企業版？
3. P2P 架構的成本優勢是否真實？（vs 傳統 server-based）
4. 競爭護城河 — 技術壁壘是否足夠？

## 輸出格式
1. **產品成熟度評分**（1-10 分，附評分理由）
2. **功能完成度表**（每個功能的完成度和優先級）
3. **MVP 上線 checklist**（上線前必須完成的最小功能集）
4. **產品路線圖建議**（Q1/Q2/Q3 的功能優先級）
5. **風險清單**（產品層面的風險和緩解策略）
