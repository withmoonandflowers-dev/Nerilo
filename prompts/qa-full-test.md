# Nerilo P2P 即時聊天平台 — 完整 QA 測試

## 你的角色
你是 Nerilo 的 QA 工程師。請對 https://nerilo.web.app 執行完整的端到端測試。
使用瀏覽器工具（Claude in Chrome）實際操作網站，不要只讀程式碼。
每個測試項目都要：實際操作 → 截圖 → 記錄結果（PASS/FAIL/BLOCKED）→ 如果 FAIL 就截 Console 錯誤。

## 測試環境準備
1. 用 tabs_context_mcp 開兩個分頁（模擬兩個使用者）
2. 分頁 A = 使用者 A（Google 登入）
3. 分頁 B = 無痕模式或不同帳號（模擬使用者 B）
4. 如果無法開兩個分頁，至少用一個分頁完整走過所有流程

## 重要技術背景
- 系統剛從 Firestore 遷移到 Firebase RTDB
- Google 登入使用 signInWithPopup（不是 redirect）
- 未登入使用者會自動匿名登入（Firebase Anonymous Auth），角色為 guest
- 訊息傳遞：P2P（WebRTC DataChannel）為主，RTDB relay 為備援
- 備援模式下，自己的訊息由 optimistic render 顯示，relay 會過濾掉 from === myUid

## 測試項目

### T1. 登入流程
- [ ] T1.1 開啟 https://nerilo.web.app → 是否正常載入？（登入頁或 Dashboard）
- [ ] T1.2 點「使用 Google 登入」→ 是否彈出 Google popup（不是 redirect）？是否能順利登入回到 dashboard？
- [ ] T1.3 登入後右上角是否顯示帳號名稱（非 guest）？
- [ ] T1.4 點登出 → 是否回到登入頁？再次進入 Dashboard 是否變回 guest？
- [ ] T1.5 用 Email/密碼登入（輸入錯誤密碼）→ 錯誤訊息是否為中文（如「電子郵件或密碼錯誤」）？
- [ ] T1.6 不輸入任何東西直接點「登入」→ 是否有驗證提示？

### T2. Dashboard（儀表板）
- [ ] T2.1 登入後 → Dashboard 是否正常顯示「我的房間」和「公開房間」？
- [ ] T2.2 點「建立新房間」→ 是否成功建立？房間是否出現在「我的房間」？
- [ ] T2.3 建立的房間是否也出現在「公開房間」？
- [ ] T2.4 Guest 使用者點「建立新房間」→ 按鈕是否 disabled 或顯示「建立房間需要登入」？

### T3. 聊天室 — 基本功能
- [ ] T3.1 點「進入」某個房間 → 聊天頁面是否正常載入？
- [ ] T3.2 輸入文字並送出 → 訊息是否出現在畫面上（右側，自己的訊息）？
- [ ] T3.3 訊息是否顯示發送時間？
- [ ] T3.4 連續快速送出多則訊息 → 是否都正常顯示？順序是否正確？
- [ ] T3.5 送出空白訊息 → 是否被擋住？
- [ ] T3.6 送出超長訊息（1000+ 字元）→ 是否正常顯示？文字是否正確 wrap？
- [ ] T3.7 **自己的訊息是否有重複顯示？**（之前有 bug：optimistic + relay 回傳導致重複）
- [ ] T3.8 點「返回」→ 是否回到 Dashboard？

### T4. 多使用者互動（核心）
- [ ] T4.1 使用者 A 建立房間，使用者 B（不同瀏覽器/帳號）進入同一房間
- [ ] T4.2 A 發訊息 → B 是否收到？延遲大約多少？
- [ ] T4.3 B 發訊息 → A 是否收到？
- [ ] T4.4 雙方同時快速發送 → 訊息順序是否正確？有無遺漏？
- [ ] T4.5 連線狀態是否顯示「備援模式」或「P2P 已連線」？
- [ ] T4.6 如果是 P2P 已連線 → 一方斷線後另一方是否偵測到？是否自動降級為備援？

### T5. Console 錯誤檢查（RTDB 遷移後重點）
- [ ] T5.1 所有頁面載入時 Console 有無紅色 error？
- [ ] T5.2 **是否有 `resource-exhausted` 錯誤？**（遷移後應該消失）
- [ ] T5.3 **是否有 `Permission denied` 錯誤？**（RTDB rules 可能遺漏）
- [ ] T5.4 是否有 `firebase/firestore` 相關錯誤？（應已完全移除）
- [ ] T5.5 送出訊息時 Console 有無錯誤？
- [ ] T5.6 是否有 CORS 相關錯誤？

### T6. 效能
- [ ] T6.1 首頁載入時間是否 < 3 秒？
- [ ] T6.2 **Bundle 大小是否減少？**（移除 firebase/firestore 後應少 ~74KB gzipped）
- [ ] T6.3 備援模式的訊息延遲是否 < 1 秒？
- [ ] T6.4 聊天室滾動是否流暢？

### T7. 安全性
- [ ] T7.1 未登入直接訪問 /dashboard → 是否正常顯示（guest 模式）？
- [ ] T7.2 未登入直接訪問 /chat/xxx → 是否能正常進入？
- [ ] T7.3 Guest 使用者是否能傳送訊息？（應該可以，因為有匿名 Firebase Auth）
- [ ] T7.4 Console 裡是否有洩漏 API key 以外的敏感資訊？

## 回報格式

### 測試總覽表

| ID | 項目 | 結果 | 備註 |
|----|------|------|------|
| T1.1 | 登入頁載入 | ✅ PASS / ❌ FAIL / ⚠️ BLOCKED | 說明 |

### 嚴重問題清單（Critical / Blocker）
列出所有 FAIL 的項目，按嚴重度排序，每個問題附上：
1. 重現步驟
2. 預期行為
3. 實際行為
4. Console 錯誤截圖
5. 建議修復方向

### 改善建議（Nice to have）
列出 UX 或效能上可以改善但不影響功能的項目。

## 重要注意事項
- 你是 QA，不是開發者。發現問題只回報，不要修程式碼。
- 每個測試步驟都要實際用瀏覽器操作，不要用「應該可以」來推測。
- 截圖是必要的證據，特別是 FAIL 的項目。
- Console 錯誤要完整複製出來，不要只說「有錯誤」。
- 如果某個測試因為前置條件不滿足而無法執行，標記為 BLOCKED 並說明原因。
