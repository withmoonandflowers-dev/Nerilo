# Spec 002：帳本賺點在型別層強制收據參照（收斂 R5）

- 軌別：feature
- 狀態：done（2026-07-14，全閘門綠：L0/L1 80/L2 1394/L3 e2e 8/8）
- 建立：2026-07-14／最後更新：2026-07-14
- 關聯：風險登記 R5、ADR-0022、ADR-0028（處置記錄）、被 Spec 001 依賴（前置，已解鎖其 T0）

## 1. 要做什麼、為什麼（specify）

稽核 R5：CreditLedger 的賺點（earn）操作在型別層不強制附收據參照，正當性目前只靠上游呼叫端自律，理論上呼叫端可繞過收據直接加點。Spec 001 的整套經濟均衡建立在「點數只能靠可驗證貢獻取得」之上，此洞不補，貨幣守恆是假的。本規格把「賺點必附已驗證的共簽收據」提升為型別層與執行期的硬約束。

**憲法檢核**：
- 目標函數加分項：補助競爭力（風險登記逐項收斂是可稽核性的實績）。
- 四條不變量影響：帳本正當性〈直接強化〉；其餘三條〈無〉。

## 2. 邊界（明確不做）

- 不改 CoSignedReceipt 密碼學格式。
- 不做跨節點帳本同步（帳本仍是本地的，Phase 1 判準不變）。
- 不處理消費側扣點（那是 Spec 001 的範圍）。

## 3. 待釐清（clarify）

（無。範圍單純，稽核建議已寫明方向：append 型別強制 receiptRef 並在內部 verify。）

## 4. 技術計畫（plan）

- CreditLedger 的 earn 型 append：型別加必填 `receiptRef`（收據雜湊/識別）＋建構時執行 `verifyReceipt` 雙簽驗證，驗證失敗拒絕入帳（fail-closed）。
- 既有呼叫端（中繼計量迴路）已持有收據，改動為傳遞參照，行為不變。
- 非 earn 型 append（初始配額、消費轉出）不受影響；型別以判別聯集區分。
